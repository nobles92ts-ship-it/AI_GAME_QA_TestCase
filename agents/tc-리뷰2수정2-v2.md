모든 답변과 보고는 한국어로 작성해.


## 작업 흐름 (2단계 — TC 1회 읽기)

### 1단계: TC 읽기 + 품질 리뷰

> ⚠️ **스냅샷 우선**: 핸드오프에 `tc_after_fix1.json` 경로가 있으면 Read 도구로 직접 읽는다. Bash로 read_gsheet_data.js 재호출 금지.

tc-리뷰.md의 **2차 리뷰 (품질 + 수정본 구조 회귀)** 규칙에 따라 진행:

- **1차 수정 반영 확인**: 이전 리뷰 파일과 1:1 대조 (반영/미반영/부분반영 명시)
- **EVAL-03**: 재현 스탭 품질 (모호한 표현, 누락 조건, 기대결과 불명확)
- **EVAL-07**: 중복 TC 검출
- **EVAL-09**: 1TC 1검증 원칙 위반
- **EVAL-11**: 취소선/추후구현 항목 TC 포함 여부 (confluence_raw.md 기반)
- **EVAL-13**: 결정 테이블 조합 커버리지

**⚠ 필수: 수정1에서 신규 추가된 TC 범위에 대한 구조 회귀 검사**
1차 수정 보고서(`review_*_v1.md` 또는 `step_result.json`)에서 `added_count`와 추가 TC ID 범위 식별.
해당 범위에 대해 아래 3개 EVAL 재실행:
- **EVAL-04 회귀**: G열(플랫폼) 값이 `PC` / `모바일` / `PC/모바일`만 있는지 확인. **긴 한글 문장이 있으면 컬럼 꼬임 CRITICAL**.
- **EVAL-05 회귀**: TC ID 연속성 (수정으로 삽입된 행의 ID가 기존과 충돌/중복 없는지)
- **EVAL-06 회귀**: 분류 그룹핑 (수정 TC가 올바른 소분류 위치에 삽입되었는지)

**2026-04-17 발생 사고 참조**: 수정1에서 신규 TC 21개 삽입 시 F열↔G열이 뒤바뀐 채 적재되었으나 2차 리뷰가 품질만 봐서 놓쳤음. 위 회귀 검사로 차단.

이슈 목록 작성 → `review_[탭명]_v2.md` 저장.

### 2단계: 수정 코드 직접 작성 → 실행

**TC를 다시 읽지 않는다.** 1단계에서 파악한 이슈를 기반으로 직접 Node.js 수정 코드를 작성하여 실행한다.

> ⚠️ **gemma4 미지원 (2026-04-20~)**: 이 파이프라인에서는 gemma4 경로를 사용하지 않는다.
> 수정 코드는 Sonnet이 직접 작성한다. 품질 편차를 없애기 위해 단일 경로로 통일.

```bash
UTIL="{WORK_ROOT}/scripts/util"
SPECS="{WORK_ROOT}/team/specs"

# 1. 리뷰 보고서의 처방(처방:)을 CRITICAL → HIGH → MEDIUM → LOW 순으로 읽는다
# 2. 각 처방을 셀 주소 기준 Node.js 수정 코드로 변환하여 직접 작성
# 3. 서식 규칙(tc-생성.md) 준수 확인 후 실행
```

> ⚠️ 작성한 코드는 실행 전 **자가 검수 필수**: 셀 주소 정확성, 이슈와 수정 내용 일치, 서식 규칙 부합.
> ⚠️ Confluence MCP 재호출 금지. 취소선 검출은 `confluence_raw.md` 파일에서 수행.

tc-수정.md의 수정 규칙에 따라 CRITICAL → HIGH → MEDIUM → LOW 순 처리.
수정 완료 후 tc-생성.md 기준 서식 적용.
