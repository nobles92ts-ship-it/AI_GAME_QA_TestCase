## 작업 흐름 (2단계 — TC 1회 읽기)

### 1단계: TC 읽기 + 품질 리뷰

> ⚠️ **스냅샷 우선**: 핸드오프에 `tc_after_fix1.json` 경로가 있으면 Read 도구로 직접 읽는다. Bash로 read_gsheet_data.js 재호출 금지.

tc-리뷰.md의 **2차 리뷰 (품질 리뷰)** 규칙에 따라 진행:

- **1차 수정 반영 확인**: 이전 리뷰 파일과 1:1 대조 (반영/미반영/부분반영 명시)
- **EVAL-03**: 재현 스탭 품질 (모호한 표현, 누락 조건, 기대결과 불명확)
- **EVAL-07**: 중복 TC 검출
- **EVAL-09**: 1TC 1검증 원칙 위반
- **EVAL-11**: 취소선/추후구현 항목 TC 포함 여부 (confluence_raw.md 기반)

이슈 목록 작성 → `review_[탭명]_v2.md` 저장.

### 2단계: 수정 실행

**TC를 다시 읽지 않는다.** 1단계에서 파악한 이슈를 기반으로 수정 코드를 작성하여 실행한다.

리뷰 보고서의 `처방:` 지시문을 읽고 직접 수정한다:
- 셀 주소/인덱스가 올바른지 확인
- 수정 내용이 리뷰 보고서 이슈와 일치하는지 확인
- 서식 규칙(tc-생성.md)에 부합하는지 확인

> ⚠️ Confluence MCP 재호출 금지. 취소선 검출은 `confluence_raw.md` 파일에서 수행.

tc-fixer.md의 수정 규칙에 따라 CRITICAL → HIGH → MEDIUM → LOW 순 처리.
수정 완료 후 tc-생성.md 기준 서식 적용.
