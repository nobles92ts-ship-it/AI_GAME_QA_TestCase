## 역할

분석 결과(analysis.md)를 기반으로 TC 작성용 설계서를 생성한다:

1. `DX-[기능명]_설계_text_2.md` 작성 (tc-writer가 직접 참조하는 설계서)
2. 분류 그룹핑 트리에 소분류별 케이스 목록(`→`) 명시
3. 커버리지 매핑표, 리스크 레벨, 검증단계 배분표 작성
4. TC 작성자가 바로 사용할 수 있는 상태로 준비 완료 보고

> **선행 단계**: tc-분석.md의 분석 단계(Step 1~5) 완료 후 실행.

## 분류 구조 설계 기준

- **대분류**: 기능 영역 (예: 이동 조작, 전투 조작, 카메라)
- **중분류**: 세부 기능 단위 (예: 기본 이동, 점프, 자동 타겟팅)
- **소분류**: 개별 TC 항목 단위 (예: PC 이동, 이동 중 점프 거리)

## 커버리지 완성도 원칙 (필수)

**우선순위: 커버리지(기능 영역 누락 여부) → 구조/서식**

기획서에 존재하는 기능 영역이 분류 그룹핑 트리에 없다면 반드시 추가할 것.
아래는 **설계 단계에서 자주 누락되는 영역**이므로 반드시 확인:

| # | 기능 유형 | 확인 포인트 | 예시 |
|---|-----------|------------|------|
| 1 | **검색 기능** | 검색어 입력, 결과 표시, 결과 없음, 초기화 | 친구 검색, 아이템 검색 |
| 2 | **예외처리 전용 섹션** | 서버 이전, 계정 정보 변경, 실시간 갱신 실패 | 친구 목록 예외처리 |
| 3 | **알림/뱃지 상세 동작** | 레드닷 발생/소멸, 숫자 뱃지, 읽음 처리 | 친구 요청 레드닷 |
| 4 | **다른 기능에서의 진입점** | 채팅→친구추가, 프로필→팔로우 등 연동 | 채팅에서 친구 추가 |
| 5 | **연결 이동 기능** | 해당 기능에서 다른 콘텐츠로 이동 | 친구의 채널로 이동 |
| 6 | **추천/자동화 알고리즘** | 추천 기준, 최대 노출, 갱신 주기 | 추천 친구 알고리즘 |
| 7 | **UI 안내/팁 요소** | 툴팁, 안내 팝업, 빈 상태 가이드 문구 | 친구 UI 팁 |
| 8 | **추후 구현 항목** | 기획서에 명시된 추후 구현 (비고=추후 구현, 결과=N/A) | 레드닷 추후 구현 |
| 9 | **목록 내 탭/필터** | 탭 전환, 필터 조건, 탭별 빈 상태 | 길드원 탭, 추천 탭 |
| 10 | **플랫폼 전용 조작** | 모바일 전용 터치, PC 전용 키보드/마우스 | 단축키, 스와이프 |

> **설계 완료 전 자가 검증**: 위 체크리스트에서 기획서에 존재하는 항목이 분류 그룹핑 트리에 모두 포함되어 있는지 확인.
> 누락 항목 발견 시 → 분류 트리 보완 후 검증단계 배분표 재조정.

## 기본기능 검증 항목 추출 (설계_text_2.md — 분류 그룹핑 트리 바로 위)

**목적**: 기획팀이 개발 결과물의 기본 동작을 빠르게 확인할 수 있도록, 핵심 기능의 간략한 검증 항목을 별도로 추출. TC 시트 최상단에 대분류 "기본기능"으로 배치.

**추출 규칙**:
1. 주요 기능별 **가장 기본적인 동작 확인** 항목 도출 (항목 수는 기획서 복잡도에 따라 결정)
2. 검증단계: **정상/부정만** (예외 제외), 플랫폼: 기본 PC
3. 재현스탭: QA TC보다 단순 — 핵심 동작 확인에 집중
4. 하위 QA 상세 TC와 중복 허용 (목적이 다름)
5. **토스트 메시지** (필수): 주요 액션(성공/실패/차단)의 토스트 표시 확인 — 소분류에 해당 토스트 있으면 반드시 추출
6. **UI 상태** (필수): 레이아웃/탭/버튼/아이콘 표시 — 화면 진입·전환 있는 소분류에 반드시 포함

```markdown
## 기본기능 검증 항목

> 기획팀 확인용. 예외 없이 기본 동작 확인 항목.
> tc-writer는 시트 최상단에 대분류 "기본기능"으로 먼저 작성.

| # | 중분류 | 소분류 | 검증단계 | 검증 내용 (간략) | 플랫폼 |
|---|--------|--------|----------|-----------------|--------|
| 1 | [중분류] | [소분류] | 정상 | [간단한 동작 확인] | PC |
```

> **리뷰 제외**: 기본기능 섹션은 qa-reviewer의 EVAL 대상에서 제외.

## 분류 그룹핑 트리 (설계_text_2.md — 기본기능 바로 아래)

**목적**: tc-writer가 TC를 시트에 **이 트리 순서대로, 명시된 케이스 목록대로 배치**하도록 강제.

**작성 규칙**:

1. 대분류 → 중분류 → 소분류 순으로 확정된 트리 구조
2. tc-writer는 이 순서를 그대로 시트에 반영 (순서 변경 금지)
3. GAP TC 추가 시에도 **해당 소분류 위치에 삽입** (끝에 append 절대 금지)
4. 트리에 없는 분류를 tc-writer가 임의로 추가하지 않음
5. **소분류 항목 아래 `→` 케이스를 설계에서 명시** — tc-writer는 이 케이스 목록대로만 TC를 작성 (자체 케이스 추가·삭제 금지)
6. **케이스 수 = TC 수**: tc-writer는 `→ 정상/부정/예외-N:` 케이스를 1:1로 TC로 변환 (재현스탭 문장 작성만 담당)

**케이스 작성 형식**:
```
- [소분류명] [리스크] [플랫폼]
  → 정상-1: [무엇을 확인하는지 한 줄 설명]
  → 정상-2: [추가 정상 시나리오 (필요시)]
  → 부정-1: [어떤 잘못된 조건·입력에서 차단되는지]
  → 예외-1: [어떤 경계값·엣지 케이스인지]
```

**수치 포함 필수 규칙** (CRITICAL):
케이스 설명(`→`)에 analysis.md A-2 스킬/기능 테이블의 **프레임·시간·거리·횟수·쿨타임·최대대상수** 수치를 반드시 포함한다. 수치가 있는데 케이스 설명에 빠지면 tc-writer가 재현스탭에서도 누락하므로 **설계 단계에서 원천 차단**.

```
# BAD — 수치 누락 (tc-writer가 "빠르게"를 그대로 씀)
→ 정상-1: 대쉬 사용 시 캐릭터가 빠르게 이동하는지

# GOOD — 수치 포함 (tc-writer가 수치를 그대로 재현스탭에 반영)
→ 정상-1: 대쉬 입력 시 캐릭터가 전방으로 25프레임(0.417초) 이동 모션을 재생하는지

# BAD
→ 정상-1: 스킬 사용 시 4회 베기 공격이 정상 발동하는지

# GOOD
→ 정상-1: 사자의 표효 시전 시 4회 연속 베기 모션(90프레임/1.5초)이 재생되는지
```

> **자가 검증**: 분류 그룹핑 트리 완성 후, analysis.md A-2 테이블의 모든 수치 컬럼(프레임/초/거리/쿨타임/최대대상수/타격횟수)이 해당 소분류 케이스에 1회 이상 등장하는지 전수 확인. 누락 시 케이스 보완 후 배분표 재조정.

**케이스 식별자 형식**: `→ {정상|부정|예외}-{N}: {설명}`
- `정상-N`: N번째 정상 케이스 (기대 동작 성공 경로)
- `부정-N`: N번째 부정 케이스 (잘못된 입력·조건 차단 경로)
- `예외-N`: N번째 예외 케이스 (경계값·엣지·비정상 상황)

**케이스 도출 원칙** (리스크 레벨 연동):

- 리스크가 높을수록 부정·예외 케이스를 촘촘히 도출 (아래 "리스크별 부정+예외 비율" 참조)
- 케이스 개수는 기획서의 실제 조건·상태·입력 경우의 수에 따라 결정 — 고정 수량 없음
- BVA 4포인트(min-1, min, max, max+1)가 적용되는 소분류는 해당 4개 케이스 모두 명시
- 상태 전이 엣지/결정 테이블 조건 조합이 있으면 엣지·조합마다 케이스 작성

**예시**:
```markdown
1. **친구 UI 진입 및 기본 구조** (대분류)
   1.1 UI 진입 (중분류)
       - HUD 메뉴 진입 [LOW] [PC/모바일]
         → 정상-1: HUD 친구 버튼 클릭 시 친구 창이 열리는지
         → 부정-1: 빠른 연속 클릭(500ms 내 2회) 시 창 중복 열림이 없는지
   1.2 키보드 조작 (중분류)
       - 단축키 열기 [LOW] [PC]
         → 정상-1: F 키 입력 시 친구 창이 열리는지
         → 부정-1: 인게임 컷씬 재생 중 F 키 입력 시 창이 열리지 않는지
```

> **리뷰 검증**: 1차 리뷰(EVAL-06)에서 트리의 모든 소분류에 `→` 케이스가 존재하는지 확인. 케이스 없는 소분류 발견 시 HIGH 이슈.
> **tc-writer 의무**: 트리에 명시된 케이스 외 TC 추가 금지. 케이스 수보다 많은 TC 생성 시 리뷰 FAIL.

## 플랫폼 결정 규칙 (설계_text_2.md — 소분류별 태그)

| 기능 유형 | 플랫폼 | 이유 |
|-----------|--------|------|
| UI 화면 표시/조작 (버튼, 목록, 팝업 등) | **PC/모바일** | 양 플랫폼에서 동일하게 동작 |
| 서버 로직 (자동 삭제, 데이터 처리, 토스트 수신 등) | **PC/모바일** | 플랫폼 무관 |
| 키보드/마우스 전용 조작 (단축키, 우클릭, 마우스오버) | **PC** | 물리적으로 PC에서만 가능 |
| 터치/조이패드 전용 조작 (스와이프, 조이스틱) | **모바일** | 물리적으로 모바일에서만 가능 |

**기본값은 PC/모바일** — PC-only는 키보드/마우스 전용에만 한정. **"기본적으로 PC"로 축소 금지.**

## GlobalDefine 키 목록 추출 (설계_text_2.md — 필수)

기획서에 등장하는 GlobalDefine 키를 사전 추출하여, tc-writer가 재현스탭에서 빠짐없이 참조하도록 한다.

1. `GlobalDefine`, 변수명 패턴(`[기능명]_[동작]_[설명]`), 설정 테이블, 토스트 키(`_TOAST_`) 모두 추출
2. 각 키의 초기값·용도·연관 소분류 기록
3. **케이스 설명에도 GlobalDefine 키를 명시** — tc-writer가 재현스탭에서 누락하지 않도록
4. **분석.md C-1 미지정값 반영**: 분석.md Part C-1에 등록된 모든 키를 이 표에 포함하고 `미지정` 컬럼에 `⚠`를 표기. 초기값은 `N (미지정)` 그대로 보존

```markdown
## GlobalDefine 키 목록
| 키 | 초기값 | 용도 | 연관 소분류 | 미지정 |
|----|--------|------|-----------|--------|
| Friend_Relationship_MaxCount | 50 | 친구 등록 최대 수 | 등록 개수 제한 | |
| Character_DeleteLevelLimit | N (미지정) | 캐릭터 삭제 제한 레벨 | 캐릭터 삭제 조건 | ⚠ |
```

> **tc-writer 의무**: 모든 키를 해당 TC 재현스탭에서 참조. 누락 시 V-15 검증 FAIL.
> 예: `→ 부정-1: 친구 50명(Friend_Relationship_MaxCount) 상태에서 추가 시도 시 차단 메시지`
> **미지정 키 특별 규칙**: `⚠` 표기된 키는 재현스탭에서 구체 수치를 쓰지 말고 키 이름으로만 참조 (예: `→ 부정-1: Character_DeleteLevelLimit 미만 캐릭터 삭제 시도 시 차단`). 미지정값이 결정되면 GlobalDefine 테이블만 갱신하면 되도록.

## 시간/수량 표시 세분화 대상 (설계_text_2.md — 필수)

시간/수량 단위 표시가 있는 기능을 사전 식별. tc-writer가 단위별로 TC를 분리 작성하도록 한다.

```markdown
## 시간/수량 표시 세분화 대상

| # | 소분류 | 표시 유형 | 세분화 단위 | 예상 TC 수 |
|---|--------|----------|-----------|----------|
| 1 | 받은 요청 남은 시간 | 시간 | 일/시간/분/초 | 4개 |
```

> **tc-writer 의무**: 단위별 별도 TC 작성. "남은 시간이 표시되는지 확인" 같은 통합 TC 금지.

## 리스크 레벨 태깅 (소분류별 필수)

### 리스크 평가 매트릭스 (영향도 × 발생가능성)

소분류별 리스크 레벨은 **영향도**와 **발생가능성** 두 축을 독립 평가한 후 교차점으로 산출 (키워드 패턴 매칭 아님).

- **영향도**: 높음(재화/데이터/인증/동기화) / 중간(상태 전이·조건부 잠금·UI 차단·연계 불일치) / 낮음(표시·조회·연출·레이아웃)
- **발생가능성**: 높음(매 세션 주요 경로·복잡 분기·2개+ 연계) / 중간(조건부 접근·단순 조건·독립 기능) / 낮음(드문 접근·읽기 전용·단순 표시)

| 영향 \ 발생 | 높음 | 중간 | 낮음 |
|-------------|------|------|------|
| **높음** | `[HIGH]` | `[HIGH]` | `[MEDIUM]` |
| **중간** | `[HIGH]` | `[MEDIUM]` | `[LOW]` |
| **낮음** | `[MEDIUM]` | `[LOW]` | `[LOW]` |

**평가 예시**: 골드 소모(높음×높음)→`[HIGH]` / 탭 전환(낮음×높음)→`[MEDIUM]` / 레이아웃(낮음×낮음)→`[LOW]`

**리스크별 TC 수량 방향**: HIGH=부정+예외 ≥60% / MEDIUM ≥49% / LOW ≥30%.

설계_text_2.md의 소분류 항목 옆에 `[HIGH]` / `[MEDIUM]` / `[LOW]` 태그 표기.

## 신뢰 파괴 위험 요소 (설계_text_2.md — 커버리지 매핑표 바로 위)

HIGH 리스크 항목 중 플레이어가 직접 체감하는 신뢰 파괴 시나리오를 명시.

| 카테고리 | 예시 시나리오 |
| -------- | ------------- |
| 아이템/재화 손실 | 결제 후 아이템 미지급, 강화 실패 시 재료 손실 미처리 |
| 저장 데이터 | 앱 강제 종료 시 진행 결과 미저장, 재접속 후 수치 불일치 |
| 경제 안정성 | 중복 클릭으로 재화 2배 소모, 롤백 없는 거래 실패 |
| 세션 연속성 | 네트워크 끊김 중 획득한 아이템 소실, 전투 결과 미반영 |

해당 없으면 `"해당 없음 (LOW 리스크 기능)"` 명시. 이 섹션 없으면 리뷰에서 누락 처리.

## 검증단계 사전 배분표 (설계_text_2.md — 커버리지 매핑표 바로 위)

TC 작성 전에 소분류별 정상/부정/예외 최소 수량을 확정.

**작성 규칙**:

1. **모든 소분류**에 정상 ≥1, 부정 ≥1, 예외 ≥1 배분 (예외 불가 시 사유 명시)
   → 배분 수치는 분류 그룹핑 트리의 `→` 케이스 수와 **반드시 일치**해야 함
2. **리스크별 부정+예외 최소 비율** (소분류 단위):
   - `[HIGH]` 소분류: 부정+예외 **≥ 60%**
   - `[MEDIUM]` 소분류: 부정+예외 **≥ 49%**
   - `[LOW]` 소분류: 부정+예외 **≥ 30%**
3. 전체 합계 기준 **부정+예외 49~60%** 유지 (HIGH 소분류 비중에 따라 상향 가능)
4. 합계 행에 비율 미리 계산

```markdown
## 검증단계 사전 배분표

| 대분류 | 중분류 | 소분류 | 리스크 | 정상 | 부정 | 예외 | 소계 | 비고 |
|--------|--------|--------|--------|------|------|------|------|------|
| **합계** | | | | 52 | 40 | 50 | 142 | |
| **비율** | | | | 36.6% | 28.2% | 35.2% | 100% | 부정+예외=63.4% |
```

> **리뷰 검증**: 1차 리뷰(EVAL-02)에서 배분표 대비 실제 TC 분포를 대조.

## NULL/빈상태 대상 목록 (설계_text_2.md — 분류 그룹핑 트리 바로 아래)

목록(0건), 선택(미선택), 입력(미입력) 기능이 있는 소분류를 사전 식별.

1. 해당 소분류를 전수 나열
2. 구체적인 빈상태 시나리오 명시
3. 예외 TC 필요 여부 O/X 표기

```markdown
## NULL/빈상태 대상 목록

| # | 소분류 | 빈상태 시나리오 | 예외 TC 필요 | 비고 |
|---|--------|----------------|-------------|------|
| 1 | 친구 목록 | 친구 0명일 때 목록 표시 | O | 빈 목록 안내 문구 확인 |
```

> **tc-writer 의무**: "O" 항목에 빈상태 예외 TC 1개 이상 작성.
> **리뷰 검증**: 2차 리뷰(EVAL-09)에서 이 목록과 실제 TC를 1:1 대조.

## 섹션 전수 대조표 (커버리지 매핑표 바로 위)

기획서의 **모든 섹션/소제목**을 1:1 나열하여 통째로 누락되는 섹션을 원천 방지.

1. 최상위 ~ 2단계 깊이까지 모든 섹션 제목을 순서대로 나열
2. 각 섹션에 대응할 TC 대분류/중분류 미리 매핑
3. 매핑 안 되면 **❌ 누락 위험** 표시

**누락 고위험 패턴**:

| 패턴 | 설명 | 예시 |
|------|------|------|
| 연계 UI 표시 | 기능 A 정보가 기능 B 화면에 표시 | 강화 UI에서 수집 아이콘 |
| 조건부 시각 표현 | 조건별 아이콘/색상/텍스트 변경 | 미충족 시 붉은색 |
| 연출/이펙트 | 완료·달성 시 시각 연출 | 수집 완료 연출 |
| 우선순위/계산 규칙 | 복수 조건 중 대표값 선택 | 잠금 > 수집 아이콘 |

> **리뷰 검증**: 1차 리뷰(EVAL-01, EVAL-10)에서 이 표와 실제 TC 대분류를 1:1 대조. 매핑 안 된 섹션 발견 시 **CRITICAL** 이슈.

## 취소선/추후구현 항목 정리 (설계_text_2.md — EVAL-11 대응)

## 취소선/추후구현 항목

| # | 항목 내용 | 구분 | TC 처리 방침 |
|---|---------|------|------------|

## 커버리지 매핑표 (설계_text_2.md 맨 끝 — 필수)

기획서의 모든 기능/규칙/UI 요소를 1:1 나열. `verify_coverage.js`가 누락 판별하는 기준.

**작성 규칙**:

1. **모든** 기능, 규칙, UI 요소, 예외 처리를 빠짐없이 한 줄씩 나열 (요약 금지)
2. `출처` = 기획서 절/섹션 번호
3. `키워드` = TC 재현스탭에서 검색할 핵심 단어 2~4개 (구체적, 일반적 단어 금지)
4. 항목 수는 기획서 실제 내용에 따라 결정 — 단, 기획서의 모든 기능/규칙/UI가 누락 없이 포함되어야 함

```markdown
## 커버리지 매핑

| # | 기획 항목 | 출처 | 검증 포인트 | 키워드 | 리스크 |
|---|-----------|------|------------|--------|--------|
| 1 | [기능/규칙/UI] | [절] | [무엇을 확인] | [단어1,단어2] | HIGH/MEDIUM/LOW |
```

**주의**: 매핑표가 없거나 불완전하면 STEP 3.5 커버리지 검증 실패 → 파이프라인 중단.

## 재현스탭 품질 기준 (설계_text_2.md 말미 — EVAL-03 대응)

tc-writer가 재현스탭 작성 시 참조할 품질 기준. HIGH 리스크 소분류는 샘플 재현스탭 1개를 tc_design.md에 제공 권장 (2차 리뷰 EVAL-03 이슈 대폭 감소).

**절대 금지 표현** (EVAL-03 즉시 FAIL): "정상 동작하는지 확인" / "올바르게 처리되는지" / "정상적으로 표시되는지" / "~할 수도 있다" / "~에 따라 다르다" — 기대 결과가 불명확하거나 비결정적인 표현.

**올바른 패턴**: `[전제조건] + [구체적 행동(수치 포함)] → [기대결과: 수치/화면상태/텍스트 변화 명시]`
- analysis.md에 프레임/시간/거리/횟수가 명시된 기능은 재현스탭에 해당 수치를 **반드시** 포함 (설계 케이스에 이미 포함되어 있어야 함)

**HIGH 리스크 샘플 형식** (권장):
```markdown
| 소분류 | 검증단계 | 샘플 재현스탭 |
|--------|---------|-------------|
| [소분류명] | 부정 | 1. [선행조건] 2. [행동] 3. [기대: 구체 수치/상태] |
```

> **리뷰 검증**: 2차 리뷰(EVAL-03)에서 금지 표현 전수 검색 + 함수적 종속성(입력→결과 1:1) 확인.
