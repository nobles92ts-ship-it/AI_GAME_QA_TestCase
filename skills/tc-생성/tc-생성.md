## 핵심 체크리스트 (작성 전 반드시 확인)

| #   | 규칙                                                | 위반 예시                                                 |
| --- | --------------------------------------------------- | --------------------------------------------------------- |
| ✅   | 1 TC = 1 검증 포인트                                | "A되고 B되는지 확인" → 분리 필수                          |
| ✅   | [사전 조건] + [행동] + [기대 결과] 한 줄 형식       | "W 키를 누르면 이동하는지 확인" (조건 없음)               |

| ✅   | 경계값 TC에 구체적 수치 필수                        | "마나가 부족할 때" → "마나 49/소모 50인 상태에서"         |

| ✅   | 정상 케이스 ≥ 40%                                   | 기본 기능 검증 최소 보장 (40% 미만 시 보강)               |
| ✅   | 소분류별 정상·부정·예외 최소 1개씩                  | 예외 케이스 유형 표 참조                                  |
| ✅   | 동일 분류 연속 배치 (흩어짐 금지)                   | groupByCategory 함수로 처리                               |
| ✅   | 분류 중복 표기 금지 (최상단 1회만)                  | renumberAndDedup 함수로 처리                              |
| ✅   | 추후 구현 기능도 TC 작성 (비고=추후 구현, 결과=N/A) | 기획서 기재 항목은 반드시 포함                            |

| B   | 대분류      | 같은 분류면 최상단 1회만                                |
| C   | 중분류      | 같은 분류면 최상단 1회만                                |
| D   | 소분류      | 같은 분류면 최상단 1회만                                |

| J   | 비고        | 왼쪽 정렬, WRAP 없음                                    |

## TC 작성 규칙

### 검증단계

| 값   | 기준                                          |
| ---- | --------------------------------------------- |
| 정상 | 기능이 의도대로 동작하는 핵심 경로            |
| 부정 | 잘못된 입력·조건에서 차단·거부되는 경로       |
| 예외 | 경계값, 동시 입력, 시스템 부하 등 엣지 케이스 |

### 재현 스탭 형식

- **형식**: `[사전 상태]에서 [행동]하면 [구체적 결과]가 발생하는지 확인`
- **사전 조건 필수**: 테스터가 시작 상태를 알 수 없으면 FAIL
- **구체적 결과 필수**: "정상 동작", "올바르게" 등 추상 표현 절대 금지
- **플랫폼 미중복**: G열에 이미 표기 → 재현 스탭에 "PC에서", "모바일에서" 시작 금지
- **1 TC = 1 검증 포인트**: "A되고 B되는지" → TC 분리

```
BAD:  "아이템이 인벤토리에 추가되고 보상 UI가 표시되는지 확인"
GOOD: TC-1 "아이템이 인벤토리에 추가되는지 확인"
      TC-2 "보상 UI가 표시되는지 확인"

BAD:  "스킬을 사용하면 정상 동작하는지 확인"
GOOD: "마나 50인 상태에서 소모량 50의 스킬을 사용하면 쿨타임 3초 UI가 표시되는지 확인"
```

### 검증단계 분포

- **부정 + 예외 49~60%** (권장 55%)
  - 49% 미만 → 엣지 케이스 부족 → 부정/예외 보강
  - 60% 초과 → 정상 경로 검증 부족 → 정상 케이스 보강
- **정상 ≥ 40%** — 기본 기능이 의도대로 동작하는지 검증하는 핵심 경로이므로 최소 보장 필수
- TC 작성 완료 후 직접 계산 → 범위 이탈 시 해당 유형 보강
- **라벨 재분류 금지**: 예외→부정 등 단순 재분류는 품질 향상 아님. 새 시나리오 없이 검증단계만 변경 금지

### 커버리지 기준 (소분류 단위)

- 정상 1개 이상 + 부정 1개 이상 + 예외 1개 이상 (적용 불가한 예외는 이유 명시 후 생략 가능)

| 기능 유형                  | 필수 예외 케이스                                                        |

| 자원 소모 (스킬·구매·사용) | 자원 정확히 0, 필요량-1 부족, 최대치 초과 시도                          |
| 상태/조건 충돌             | 중복 실행, 선행 조건 없이 후속 동작, 복수 버프/디버프 동시 적용         |

| 멀티플레이 (해당 기능)     | 동시 행동 충돌, 권한 경쟁, 연결 해제 중 상호작용, 고레이턴시            |

## 기본기능 섹션 (시트 최상단 배치)

**목적**: 기획팀이 개발 결과물의 기본 동작을 확인하기 위한 섹션. QA 상세 TC 위에 배치.

**작성 규칙**:

1. **대분류**: "기본기능" 고정
2. **검증단계**: 정상/부정만 (예외 제외)
3. **재현스탭**: 단순하게 — 핵심 동작 확인에 집중 (경계값, 복잡한 조합 불필요)
4. **모바일 결과**: 기본값 N/A
5. **TC ID**: 001부터 시작 (QA TC와 통합 채번)
6. **소스**: text_2.md의 "기본기능 검증 항목" 참조
7. **하위 QA TC와 중복 허용** — 목적이 다르므로 문제 없음

**작성 순서**:

```
Step 1: 기본기능 tc() 호출 (text_2.md의 기본기능 검증 항목 기반)
Step 2: QA 상세 TC tc() 호출 (기존 방식)
```

**리뷰 제외**: 기본기능 섹션은 qa-reviewer의 EVAL 대상에서 제외된다.

## 분류 그룹핑

- 동일 대/중/소분류는 **반드시 연속 배치** (흩어짐 금지)
- 분류 중복 표기: 최상단 1회만, 이후 같은 분류 행은 빈칸
- `groupByCategory` → `renumberAndDedup` 순으로 처리

## 플랫폼 적용 규칙 (필수 — tc-설계의 결정을 따름)

**플랫폼 결정은 tc-설계(설계 단계)에서 수행**하며, text_2.md의 분류 그룹핑 트리에 소분류별 `[PC/모바일]` / `[PC]` / `[모바일]` 태그로 확정되어 있다.

tc-writer는 **text_2.md의 플랫폼 태그를 그대로 G열에 반영**한다.

| 상황                                | 처리                                                |
| ----------------------------------- | --------------------------------------------------- |
| text_2.md에 플랫폼 태그 있음        | 해당 태그를 그대로 사용                             |
| text_2.md에 플랫폼 태그 없음 (누락) | **PC/모바일을 기본값으로 사용** (PC-only 축소 금지) |

| 플랫폼 값 | 결과 열 초기값      |
| --------- | ------------------- |
| PC/모바일 | H·I 모두 `미진행`   |
| PC        | H=`미진행`, I=`N/A` |
| 모바일    | H=`N/A`, I=`미진행` |

## 플랫폼 분리

- PC와 모바일(조이패드) 조작이 다른 경우 **별도 행**으로 분리
- 해당 없는 결과 열 → `N/A`, 해당 있는 결과 열 → `미진행`

## 추후 구현 항목

- 기획서에 "추후 구현", "미구현", "추후 적용 예정" 등으로 표기된 항목도 **TC 반드시 작성**
- 비고(J열): `추후 구현` / 결과 열(H·I): `N/A`

## GlobalDefine 참조 규칙 (필수)

기획서에 GlobalDefine 키(예: `Friend_Relationship_MaxCount`, `Friend_TOAST_Relationship_Login`)가 명시된 경우:

1. **재현스탭에 GlobalDefine 키 반드시 포함**: `친구가 50명(최대값, Friend_Relationship_MaxCount)인 상태에서...`
2. **수치가 있으면 구체적으로 명시**: `24시간(Friend_Relationship_ResponseTime, 초기값 24시간)이 경과하면...`
3. **토스트 메시지는 키와 함께**: `"{0}님과 친구가 되었습니다." (Friend_TOAST_Relationship_Accept) 토스트가 표시되는지 확인`

> V2 품질 기준: GlobalDefine 키가 있는 기능은 해당 키를 재현스탭에서 참조하지 않으면 리뷰에서 FAIL 처리.

## 시간/수량 표시 세분화 규칙

기획서에 시간 표시(남은 시간, 경과 시간 등) 또는 수량 단위 표시가 있는 경우, **각 단위별로 별도 TC를 작성**한다:

```
예: "남은 시간 표시" 기능
TC-1: 남은 시간이 1일 이상일 때 "N일 남음" 형식으로 표시되는지 확인
TC-2: 남은 시간이 1시간 이상 24시간 미만일 때 "N시간 남음" 형식으로 표시되는지 확인
TC-3: 남은 시간이 1분 이상 1시간 미만일 때 "N분 남음" 형식으로 표시되는지 확인
TC-4: 남은 시간이 1분 미만일 때 "N초 남음" 형식으로 표시되는지 확인
```

> 통합 1건("남은 시간이 표시되는지 확인")으로 작성하면 리뷰에서 FAIL.

## 서식 규칙

**서식은 `apply_format_tab.js`가 단일 소스(Single Source of Truth)**이며, TC 데이터 생성 후 스크립트를 실행하면 자동 적용된다.

스크립트가 처리하는 항목: 헤더 서식, 데이터 행 정렬/WRAP, 검증단계 조건부 서식(정상/부정/예외), 결과 열 조건부 서식(PASS/FAIL/BLOCK/N/A), 드롭다운, 필터, 열 너비, 빈 행 정리.

프로젝트 정보 패널(K~L열)은 tc-팀-v2 완료 처리 단계에서 `add_project_info.js`가 일괄 적용. tc-writer-v2는 실행 금지.

## 작업 흐름

1. MD 파일 경로를 받으면 파일 내용 읽기
2. 기능 구조(대/중/소분류) 파악
3. **기본기능 TC 먼저 도출**: text_2.md의 "기본기능 검증 항목" 참조하여 기본기능 TC 작성
4. 각 기능별 QA 상세 TC 케이스 도출 (검증단계별로 분류):
   - 정상 동작 케이스
   - 부정 케이스 (잘못된 입력, 차단 확인)
   - 예외/경계값 케이스
5. 플랫폼별 행 분리 적용
6. **분류 그룹핑 필수**: 동일 대분류/중분류/소분류는 반드시 한 곳에 모아서 연속 배치
7. **분류 중복 표기 제거**: 최상단 1회만 표기, 이후 행은 빈칸
8. **마스터 스프레드시트에 새 탭으로 생성** (`create_gsheet_tc.js` 템플릿 기반, 별도 파일 생성 금지)

   **청크 분할 작성 (필수)**: tc() 호출 코드를 전체 한 번에 작성하지 않고 **대분류 단위**로 나누어 순차 작성한다.

   ```
   Step A: JS 파일 헤더 작성 (imports, tc 함수 정의, 변수 초기화)
   Step B: 대분류별 순차 작성
           한 대분류의 tc() 호출을 모두 작성 → 파일에 추가(append) → 다음 대분류
           각 대분류 시작 시 주석: // ── 대분류: [이름] ──
   Step C: 모든 대분류 완료 후 JS 파일 푸터 작성 (업로드 실행부)
   Step D: 최종 스크립트 한 번 실행
   ```

   > 전체 TC를 한 번에 작성하면 컨텍스트 부하로 속도 저하 및 누락 발생.

    cd "{PROJECT_ROOT}"
    node -e "
    const {google}=require('googleapis'); const {getAuthClient}=require('./scripts/util/google_auth');
    const XLSX=require('xlsx');
    (async()=>{
      const auth=await getAuthClient();
      const sheets=google.sheets({version:'v4',auth});

## 완료 전 자가 검증 (verification-loop)

TC 데이터 생성 + 서식 적용 완료 후, 완료 보고 전 아래 항목을 **반드시 직접 확인**한다.
하나라도 FAIL이면 즉시 수정 후 재확인. 전부 PASS일 때만 완료 보고 가능.

```
VERIFICATION REPORT
===================
[ ] V-01 컬럼 10개 (A~J) 확인
[ ] V-02 TC ID: =TEXT(ROW()-1,"000") 수식 입력, 001부터 연속 표시, 중복 없음
[ ] V-03 플랫폼 값: PC / 모바일 / PC/모바일 만 사용
[ ] V-04 검증단계 값: 정상 / 부정 / 예외 만 사용
[ ] V-05 재현 스탭: "정상 동작", "정상적으로" 등 추상 표현 없음 / 플랫폼·사전상태·기대결과 3요소 포함
[ ] V-06 분류 그룹핑: 동일 소분류 TC 연속 배치 (분산 없음)
[ ] V-07 분류 표기: 동일 분류 최상단 1회만, 이후 행 빈칸
[ ] V-08 서식 적용: apply_format_tab.js 실행 완료
[ ] V-09 플랫폼별 결과 열 초기값: PC→I열 N/A, 모바일→H열 N/A, PC/모바일→H/I 미진행
[ ] V-10 검증단계 분포: 부정+예외 49~60% (권장 55%), 정상 ≥ 40% — ※ 기본기능 섹션 제외하고 QA 상세 TC만 집계
[ ] V-11 플랫폼 결정: UI/서버 기능은 PC/모바일이 기본값. PC-only는 키보드/마우스 전용에만 한정
[ ] V-12 GlobalDefine 참조: 기획서에 GlobalDefine 키가 있는 TC의 재현스탭에 해당 키가 포함되어 있는지 확인
[ ] V-13 시간/수량 세분화: 시간 표시, 수량 단위 등 표시 규칙이 있는 기능에서 단위별 TC 분리 여부

Overall: PASS (13/13) → 완료 보고 진행
         FAIL (N/13) → 해당 항목 수정 후 재검증
```

검증 결과는 완료 보고서의 "자가 검증" 항목에 포함한다.
