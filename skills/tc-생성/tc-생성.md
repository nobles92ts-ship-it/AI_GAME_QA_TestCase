# TC 생성 운용 지침

> **목적**: tc-설계 단계에서 확정된 설계 내역(tc_design.md)을 읽어 TC 양식(10컬럼)에 맞게 스프레드시트로 코딩한다. 분석/설계 판단은 하지 않으며, 설계 결정을 그대로 옮기는 역할.

모든 답변과 보고는 **한국어**로 작성한다.

## 핵심 체크리스트 (작성 전 반드시 확인)

| #   | 규칙                                                | 위반 예시                                                 |
| --- | --------------------------------------------------- | --------------------------------------------------------- |
| ✅   | 1 TC = 1 검증 포인트                                | "A되고 B되는지 확인" → 분리 필수                          |
| ✅   | (사전 상태)에서 (행동)하면 (구체적 기대 결과)가 되는지 확인 — 개행 없는 단일 서술문 | "W 키를 누르면 이동하는지 확인" (조건 없음) |
| ✅   | 추상적 표현 금지 ("정상 동작", "정상적으로" 등)     | "정상 동작하는지 확인" → FAIL                             |
| ✅   | 재현 스탭에 플랫폼 중복 기재 금지                   | "PC/모바일에서 ~하면" → FAIL                              |
| ✅   | 경계값 TC에 구체적 수치 필수                        | "마나가 부족할 때" → "마나 49/소모 50인 상태에서"         |
| ✅   | 부정 + 예외 합계 49~60% (권장 55%)                  | 49% 미만 → 엣지 부족, 60% 초과 → 정상 경로 부족           |
| ✅   | 정상 케이스 ≥ 40%                                   | 기본 기능 검증 최소 보장 (40% 미만 시 보강)               |
| ✅   | 소분류별 정상·부정·예외 최소 1개씩                  | 예외 케이스 유형 표 참조                                  |
| ✅   | 동일 분류 연속 배치 (흩어짐 금지)                   | groupByCategory 함수로 처리                               |
| ✅   | 분류 중복 표기 금지 (최상단 1회만)                  | renumberAndDedup 함수로 처리                              |
| ✅   | 추후 구현 기능도 TC 작성 (비고=추후 구현, 결과=N/A) | 기획서 기재 항목은 반드시 포함                            |
| ✅   | Output Format 템플릿 잔류 금지                      | `Output Format: '[사전 상태]에서...'` → 완성 문장 필수    |
| ✅   | 수식 대신 직접값 입력                               | INDIRECT() 등 수식 → #ERROR! 원인. JSON 기반이므로 불필요 |
| ✅   | 추가 케이스는 기존 그룹 내 삽입                     | 시트 끝에 append → 분류 분산 CRITICAL 유발               |
| ✅   | HIGH 리스크 소분류 부정+예외 ≥ 60%                  | 58.3% → EVAL-02 FAIL                                     |

## 컬럼 구조 (10컬럼, 순서 엄수)

| 열  | 컬럼명      | 값 규칙                                                 |
| --- | ----------- | ------------------------------------------------------- |
| A   | TC ID       | `=TEXT(ROW()-1,"000")` 수식 — 행 삽입/삭제 시 자동 갱신 |
| B   | 대분류      | 같은 분류면 최상단 1회만                                |
| C   | 중분류      | 같은 분류면 최상단 1회만                                |
| D   | 소분류      | 같은 분류면 최상단 1회만                                |
| E   | 검증단계    | 정상 / 부정 / 예외                                      |
| F   | 재현 스탭   | (사전 상태)에서 (행동)하면 (구체적 기대 결과)가 되는지 확인 — 개행 없는 단일 서술문, 왼쪽 정렬 |
| G   | 플랫폼      | PC / 모바일 / PC/모바일                                 |
| H   | PC 결과     | PASS / FAIL / BLOCK / 미진행 / N/A                      |
| I   | 모바일 결과 | PASS / FAIL / BLOCK / 미진행 / N/A                      |
| J   | 비고        | 왼쪽 정렬, WRAP 없음                                    |

### tc_data.json 행 형식 (writer 공통 — 순서 엄수)

모든 writer(haiku_tc_writer.py 포함)가 생성하는 JSON 배열 각 행은 **7개 문자열**:

```
["대분류", "중분류", "소분류", "검증단계", "재현스탭", "플랫폼", "비고"]
// idx 0     idx 1     idx 2    idx 3     idx 4 ⚠    idx 5 ⚠  idx 6
//   B         C         D        E         F ⚠        G ⚠      J
```

> **⚠ CRITICAL — 순서 엄수 (2026-04-17 재발 사고)**:
> 반드시 **재현스탭(index 4) → 플랫폼(index 5)** 순.
> 반대로 쓰면 시트에 F열=플랫폼, G열=재현스탭으로 뒤바뀜.
> 과거 사고: 2026-04-17 아바타_탈것_합성_확정_시스템 TC 176~196(21개)에서 컬럼 꼬임 발생. 2차 리뷰까지 놓쳐 배포됨.

**배열 조립 직전 자가 점검 (의무)**:
- [ ] index 3 위치 값이 **`정상` / `부정` / `예외`** 중 하나인가? (검증단계)
- [ ] index 4 위치 값이 **긴 한글 서술문**인가? (재현스탭 — "~하면 ~되는지 확인")
- [ ] index 5 위치 값이 **`PC` / `모바일` / `PC/모바일`** 중 하나인가? (플랫폼)
- [ ] index 6 위치 값이 **짧은 메모** 또는 **빈 문자열**인가? (비고)
- 위 4개 중 **하나라도 어긋나면 배열 즉시 수정 후 쓰기**. 이 점검 없이는 write 호출 금지.

**BAD 예시** (컬럼 꼬임):
```json
["전투", "스킬", "대쉬", "정상", "PC", "대쉬 시 25프레임 이동", ""]
//                           ↑ index 4=PC (틀림)   ↑ index 5=서술문 (틀림)
```

**GOOD 예시**:
```json
["전투", "스킬", "대쉬", "정상", "대쉬 시 25프레임 이동", "PC", ""]
//                           ↑ index 4=서술문 ✓    ↑ index 5=PC ✓
```

## TC 작성 규칙

### 검증단계

| 값   | 기준                                          |
| ---- | --------------------------------------------- |
| 정상 | 기능이 의도대로 동작하는 핵심 경로            |
| 부정 | 잘못된 입력·조건에서 차단·거부되는 경로       |
| 예외 | 경계값, 동시 입력, 시스템 부하 등 엣지 케이스 |

### 재현 스탭 형식

- **단위**: 개행 없는 단일 문자열. 사전 상태·행동·기대 결과 세 요소를 하나의 서술문 안에 인라인으로 통합한다.
- **문장 구조**: `(사전 상태)에서 (행동)하면 (구체적 기대 결과)가 되는지 확인`
- **사전 조건 필수**: 테스터가 시작 상태를 알 수 없으면 FAIL
- **구체적 결과 필수**: "정상 동작", "올바르게" 등 추상 표현 절대 금지
- **플랫폼 미중복**: G열에 이미 표기 → 재현 스탭에 "PC에서", "모바일에서" 시작 금지
- **1 TC = 1 검증 포인트**: "A되고 B되는지" → TC 분리

```
BAD:  "아이템이 인벤토리에 추가되고 보상 UI가 표시되는지 확인"
GOOD: TC-1 "아이템이 인벤토리에 추가되는지 확인"
      TC-2 "보상 UI가 표시되는지 확인"

BAD:  "스킬을 사용하면 정상 동작하는지 확인"
GOOD: "마나 50인 상태에서 소모량 50의 스킬을 사용하면 쿨타임 3초 UI가 표시되는지 확인"

BAD:  "[사전조건] 마나 50 [행동] 스킬 사용 [기대결과] 쿨타임 표시"
      → 대괄호 태그 분리 표기 금지. 하나의 서술문으로 통합할 것.
GOOD: "마나 50인 상태에서 스킬을 사용하면 쿨타임 3초 UI가 표시되는지 확인"
```

### 커버리지 기준 (소분류 단위)

- 정상 1개 이상 + 부정 1개 이상 + 예외 1개 이상 (적용 불가한 예외는 이유 명시 후 생략 가능)

## 기본기능 섹션 (시트 최상단 배치)

**목적**: 기획팀이 개발 결과물의 기본 동작 확인 섹션. QA 상세 TC 위에 배치.

> **⚠ 2026-04-24 변경**: 기본기능도 **분류 체계는 일반 TC와 동일하게 통일**.
> 더 이상 대분류 "기본기능" 고정 사용 금지. 자동화 사전조건 계층(대=카테고리, 중=뎁스, 소=현재 화면)을 그대로 따른다.
> 기본기능 식별은 **회색 배경(#D9D9D9) + J열 `기본기능` 태그**로 처리.

**작성 규칙** ⚠ **[CRITICAL] 위반 시 리뷰 자동 재작업**:

1. **대/중/소분류**: 일반 TC와 **동일한 자동화 사전조건 계층** 사용 (대=카테고리/이전 화면, 중=뎁스/셋업, 소=현재 화면). tc-설계.md "분류 구조 설계 기준" 참조
2. **식별자**: J열에 `기본기능` 태그 + 행 배경 #D9D9D9 (자동 적용)
3. **검증단계**: 정상/부정만 (예외 제외)
4. **재현스탭**: 단순하게 — 핵심 동작 확인 집중 (경계값, 복잡 조합 불필요). **단, GlobalDefine 키와 기획서 안내 문구는 반드시 인용**
5. **플랫폼(G열)**: **`PC` 고정** (모바일 N/A 정책 때문)
6. **모바일 결과(I열)**: **`N/A` 고정** (절대 `미진행` 쓰지 말 것)
7. **GlobalDefine 키 필수 포함**: 기획서에 GlobalDefine 키(예: `Avatar_UI_Compound_Register_Info`)가 있으면 F열 재현스탭에 `(키명)` 형태로 반드시 인용
8. **기획서 안내 문구 인용**: 기획서에 정의된 안내 문구가 있으면 큰따옴표로 그대로 인용
9. **TC ID**: 001부터 시작 (QA TC와 통합 채번)
10. **소스**: text_2.md의 "기본기능 검증 항목" 참조
11. **하위 QA TC와 중복 허용** — 목적이 다르므로 문제 없음. 분류가 같아도 OK (자동화 fixture 공유 가능)

**BAD vs GOOD 예시** (2026-04-24 분류 체계 통일 적용):

```
❌ BAD (구 규칙 — 분류 예외 사용):
  대분류: 기본기능
  중분류: (없음 또는 기능명)
  소분류: 합성 화면 진입
  G: PC             I: N/A
  → 대분류 "기본기능" 고정 — 자동화 사전조건 계층과 불일치

✅ GOOD (신규 — 통일 분류):
  대분류: 아바타/탈것 합성              ← 일반 QA TC와 동일 카테고리
  중분류: 합성 Tab 진입                  ← 셋업
  소분류: 합성 재료 등록 화면            ← 현재 화면
  G: PC             I: N/A
  J: 기본기능
  F: "합성 Tab에 최초 진입한 상태에서 합성 Tab을 선택하면 \"오른쪽 목록에서 합성할
     아바타를 선택해주세요.\" (Avatar_UI_Compound_Register_Info) 기본 안내 문구가
     표시되는지 확인"
```

**작성 순서**:

```
Step 1: 기본기능 tc() 호출 (text_2.md의 기본기능 검증 항목 기반, 분류는 신규 체계)
Step 2: QA 상세 TC tc() 호출 (분류는 신규 체계 동일)
```

**리뷰 제외**: 기본기능 섹션(J열 `기본기능` 태그 행)은 qa-reviewer의 EVAL-02/EVAL-09 등 분포 비율 계산에서 제외. 단 EVAL-19(분류 계층 적합성)는 동일 적용.

**행 배경색**: J열 `기본기능` 태그 행의 B~F열 배경색 **연한 회색3** (#D9D9D9) 적용.
- `create_gsheet_tc_from_json.js`에서 TC 업로드 시 자동 적용 (별도 작업 불필요)
- 식별 기준 변경: 과거 "대분류=기본기능" → 신규 **"J열=기본기능"**

## 분류 그룹핑

- 동일 대/중/소분류는 **반드시 연속 배치** (흩어짐 금지)
- 분류 중복 표기: 최상단 1회만, 이후 같은 분류 행은 빈칸
- `groupByCategory` → `renumberAndDedup` 순 처리

### 분류 계층 의미 (자동화 사전조건 계층) ⚠ [CRITICAL]

분류는 **단순 기능 그룹핑이 아닌 자동화 셋업 단계**다. tc-설계.md "분류 구조 설계 기준" 참조 (SSoT).

| 분류 | 의미 | F열에 들어가는가? |
|---|---|---|
| 대분류 | 카테고리 / 이전 화면 / 최상위 셋팅 | ❌ (fixture 셋업) |
| 중분류 | 뎁스 / 중간 셋팅 (예: 신규 계정, 인게임-UI 진입) | ❌ (셋업) |
| 소분류 | **현재 화면** (재현스탭 실행 직전 도달 화면) | ❌ (네비게이션) |
| F열 | 소분류 화면 위에서 가능한 **액션 + 검증** | ✅ |

**writer 작성 규칙**:
1. tc-designer가 결정한 분류 트리를 **그대로 따르되**, F열 작성 시 **진입 동작 중복 기재 금지**
2. 소분류가 "재료 등록 화면"이면 → F열에 "HUD에서 합성 진입 후 ~" 같은 진입 동작 작성 금지
3. F열은 **그 화면 도달 이후의 액션만** 기술

```
❌ BAD (진입 동작 중복):
  D(소분류): 재료 등록 화면
  F: "HUD에서 합성 버튼을 클릭하고 합성 Tab에 진입하여 재료 등록 화면에서
      재료 슬롯을 클릭하면 보유 재료 목록이 표시되는지 확인"

✅ GOOD (화면 위 액션만):
  D(소분류): 재료 등록 화면
  F: "재료 등록 화면에서 재료 슬롯을 클릭하면 보유 재료 목록이 표시되는지 확인"
```

## C~F열 검증 규칙 (writer/fixer/updater 공통) ⚠ [CRITICAL]

**단일 소스**: 본 섹션 = SSoT. 에이전트는 시트 적재 시 반드시 `validate_tc_rows.js`의 두 함수 호출.

### 검증 함수

```js
const { validatePreWrite, validatePostWrite, formatViolations } =
  require('{WORK_ROOT}/scripts/util/validate_tc_rows.js');
```

### 적재 직전 (validatePreWrite)

| 항목 | 컬럼 | 위반 시 | 근거 |
|---|---|---|---|
| 빈 값 금지 | F (재현스탭) | CRITICAL | 모든 행 필수 |
| 동사·동작 표현 금지 | C/D/E (대/중/소) | HIGH | EVAL-19 ①② |
| 진입 동작 중복 금지 | F (재현스탭) | HIGH | EVAL-19 ③ (`HUD에서`, `~Tab을 클릭`, `~화면 진입 후 ~화면에서`) |

### 적재 직후 (validatePostWrite — Read-Back)

| 항목 | 컬럼 | 위반 시 | 근거 |
|---|---|---|---|
| 행 수 일치 | 전체 | CRITICAL | 병합셀로 인한 행 손실 차단 |
| C/D/E/F 값 일치 | C~F | CRITICAL | 컬럼 꼬임 차단 |
| F→G 침범 차단 | G | CRITICAL | 2026-04-17 사고 (긴 한글 문장이 G열에 적재됨) |

### 호출 의무

- writer: 모든 적재에 pre + post **둘 다** 호출
- fixer: 신규 행 있으면 pre, 모든 수정에 post
- updater: pre + post 둘 다

검증 실패 = 즉시 STOP. LLM 판단으로 skip 금지. 다음 STEP으로 넘어가지 않는다.

## F열 내부 설계 태그 인용 금지 ⚠ [CRITICAL]

설계 산출물(`tc_design.md`)의 **상태표·결정표·문의 태그**는 설계자가 추적용으로 쓰는 것. **재현스탭(F열)에 그대로 옮겨 적으면 QA·기획자에게 의미 불명**.

### 금지 태그 패턴

| 패턴 | 의미 (설계 내부) | 시트에 남기면 |
|---|---|---|
| `(ST-N #M)` | 상태 전이표 N의 M번 행 | ❌ QA가 알 수 없음 |
| `(DT-N)` / `(DT-N #M)` | 결정 테이블 N | ❌ |
| `(C-N #M)` | 기획 확인 필요 항목 N의 M번 | ❌ (별도로 J="기획 확인 필요" 부여) |
| `(B-N)` | 비즈니스 규칙 N | ❌ |
| `(§N-M)` | 기획서 섹션 번호 | ❌ (소분류로 표현) |
| `` `변수명`(미지정) `` | 미지정 GlobalDefine | ❌ (J="기획 확인 필요"로 분리) |

### BAD vs GOOD (2026-04-24 아이템_수집_개선 회귀 사례)

```
❌ BAD:

      해제되며 수집 효과 출력으로 복귀(ST-2 #2)되는지 확인"
  → "(ST-2 #2)" 가 QA에겐 무의미. 설계자만 아는 표기

✅ GOOD:
  F: "수집 항목 선택(세부 정보 출력) 상태에서 세부 정보 닫기 버튼을 입력하면
      세부 정보 해제 + 항목 선택 해제 + 수집 효과 출력 화면으로 복귀하는지 확인"
  → 상태 전이 결과를 자연어로 풀어 씀 (해제+해제+복귀)
```

```
❌ BAD:
  F: "...List에서 아이콘 클릭 시 하이라이트 연출이 `하이라이트 연출 재생 시간`(미지정)
```

### 처리 규칙

1. tc-writer-v2가 tc_design.md → 시트 작성 시 **내부 태그는 의미만 추출**해서 자연어로 풀어 쓸 것
2. 태그가 **설명용 보조**(괄호 부속)면 → 괄호째 제거
3. 태그가 **기획 미정 신호**(`C-N #M` 패턴, `(미지정)` 등)면 → 태그 제거 + **J열 `기획 확인 필요`** 부여 + H/I `미진행`

## F열 "또는" 표현 규칙 (사전조건·결과 분기 금지) ⚠ [CRITICAL]

**한 TC = 사전조건 1개 + 과정 1개 + 기대결과 1개**. "또는"으로 분기되면 1NF 위반.

### 금지 패턴 3종

**① 결과 분기** (가장 심각):
- ❌ "로딩 화면 **또는** 오류 안내가 표시되는지" — 둘 중 뭐가 맞는지 미정
- ❌ "전체 롤백 **또는** 전체 적용" — 원자성 정책 미정

**② 사전조건 분기**:
- ❌ "합성 실패 **또는** 성공-고급 이하 결과물 클릭 시 → 1회 회전" — 두 사전조건이 다른 케이스
- ❌ "미획득 **또는** 1개 획득 아바타가 숨겨지는지" — 두 사전조건은 별도 TC

**③ 과정 분기**:
- ❌ "Tab 선택 **또는** 재료 등록 시" — 두 행동은 별도 TC

### 처리 규칙

1. **기획서로 결정 가능** → **두 TC로 분리** (TC-A, TC-B)
2. **기획서로 판단 불가** → J열에 **`기획 확인 필요`** 부여 + H/I = `미진행`

### 허용 패턴 (분기 아님)

- "확정 완료 **또는** 확정 대기 중 하나로만 존재" — 결과 1개("정합성 유지") + 허용 상태 나열 (○ — 단, "허용 상태 명세를 기획에서 확정한 경우"에만)
- 그 외 모든 "또는" → 의심 → 기획 확인 필요 또는 분리
