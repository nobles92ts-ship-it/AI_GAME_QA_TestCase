# TC 리뷰 운용 지침

모든 답변과 보고는 **한국어**로 작성한다.

---

## TC 데이터 읽기

TC 데이터는 **Google Sheets MCP로 직접 읽고, 기획서(tc_design.md)를 반드시 병행 참조**한다.

```
1. sheet_info.txt에서 SHEET_ID, TAB_NAME 확인
2. mcp__google-sheets__get_sheet_data 로 시트 직접 읽기:
   - spreadsheet_id: SHEET_ID
   - range: "[TAB_NAME]!A:J"
3. specs 폴더의 tc_design.md를 읽어 기획서 구조 파악:
   - 경로: {WORK_ROOT}/team/specs/[기능명]/tc_design.md
   - 대/중/소분류 목록, 암묵적 요구사항, 리스크 레벨, 커버리지 매핑표 확인
4. TC와 기획서를 병행 비교하며 리뷰 진행
   - "기획서의 이 항목이 TC에서 실제로 테스트되고 있는가?" 관점 유지
   - 커버리지 매핑표의 키워드를 TC 재현스탭에서 검색하여 커버 여부 확인
5. **EVAL-10 기획서 원문 접근** (섹션 전수 대조용):
   - sheet_info.txt에서 CONFLUENCE_URL 확인
   - `mcp__claude_ai_Atlassian__getConfluencePage`로 기획서 원문의 섹션 목차 직접 추출
   - tc_design.md가 아닌 **기획서 원문 기준**으로 TC 대분류와 1:1 매핑하여 누락 섹션 검출
   - Confluence URL 없으면 specs 폴더의 analysis.md를 원문 대용으로 사용
6. **EVAL-11 취소선/추후구현 접근** (교차 검증용):
   - `mcp__claude_ai_Atlassian__getConfluencePage` (contentFormat: `adf`)로 기획서 원문 읽기
   - ADF에서 `"type":"strike"` mark가 있는 텍스트 노드를 모두 추출
   - "추후구현", "추후 적용", "미구현", "TBD", "추후 개발" 등 보류 표현을 본문에서 검색
```

## 기본기능 섹션 리뷰 제외

대분류가 **"기본기능"**인 TC 행은 **모든 EVAL 검토 대상에서 제외**한다.

- **EVAL 01~13 전부**에서 기본기능 행은 무시 (01~11 + 상태 전이 12 + 결정 테이블 13)
- TC 수 집계, 검증단계 비율 계산(부정+예외 49~60% 등)에서도 **제외** — EVAL-02 분모에서 기본기능 TC를 뺀 뒤 비율 산출
- 기본기능 행의 TC ID 채번은 전체 통합이므로, TC ID 연속성(EVAL-05) 검증 시 기본기능 행의 ID 갭은 허용

> 기본기능 섹션은 기획팀 확인용이며, QA 리뷰 품질 기준과는 별도로 관리된다.

## 차수별 리뷰 역할 (역할 분리)

각 차수는 **명확히 분리된 역할**을 가진다. 자기 차수의 담당 EVAL만 검토하고, 담당 외 EVAL은 보지 않는다.
이전 리뷰 파일이 있으면 반드시 먼저 읽고 수정 반영 여부를 확인한 뒤 추가 검토.

### 1차 리뷰 — 구조 리뷰 ("빠진 게 없는가?")

**담당 EVAL: 01, 02, 04, 05, 06, 08, 10**

커버리지와 구조적 정합성에만 집중한다. 재현 스탭 품질(03), 중복(07), 1TC 규칙(09)은 **보지 않는다**.

- **EVAL-01**: 기획서(tc_design.md) 대비 TC 커버리지 전수 확인 (대분류/중분류/소분류 빠짐없이)
- **EVAL-02 (기본기능 제외)**: 각 소분류에 정상+부정+예외 최소 1개씩 존재하는지 확인. **추가로 tc_design.md의 [검증단계 사전 배분표]와 실제 TC 분포를 대조** — 배분표 최소 수량 미달 소분류가 있으면 **HIGH** 이슈. **리스크 기반 비율 확인 (기본기능 TC 제외한 분모로 산출)**: `[HIGH]` 소분류 부정+예외 < 60% → **HIGH**, `[MEDIUM]` < 49% → **MEDIUM**, `[LOW]` < 30% → **LOW** 이슈
- **EVAL-04**: 플랫폼 값(G열)이 PC / 모바일 / PC/모바일 외 값 없는지 확인
- **EVAL-05**: TC ID가 001부터 순서대로, 중복·빠짐 없는지 확인
- **EVAL-06**: 동일 소분류 TC가 연속 배치되어 있는지 (분산 금지). **추가로 tc_design.md의 [분류 그룹핑 트리] 순서와 실제 시트 배치 순서를 대조** — 트리 순서 위반 시 **CRITICAL** 이슈
- **EVAL-08**: 동일 대/중/소분류 최상단 1회만 표기, 이후 빈칸인지 확인
- **EVAL-10**: 기획서 원문(Confluence/analysis.md)의 **모든 섹션 제목**을 나열하고, TC 대분류와 1:1 매핑하여 누락 섹션 0건인지 확인. 누락 발견 시 **CRITICAL** 이슈. 추가로 tc_design.md의 **[기능 간 관계]** 테이블이 있으면, 교차 TC 필요("O") 항목이 TC에 반영되었는지 확인 (미반영 시 **HIGH**)
- **EVAL-12**: 상태 전이 TC 커버리지 — tc_design.md의 **[상태 전이 테이블]**이 있는 소분류를 대상으로, 모든 전이 엣지(화살표)에 TC가 존재하는지 확인. 비정상 전이(동일 상태에서 동일 이벤트 재발생) TC 누락 시 **HIGH** 이슈. 상태 전이 테이블이 tc_design.md에 없으면 상태 있는 소분류(쿨타임, 버프, 잠금 등) 직접 식별 후 전이 커버 여부 확인
- 누락 케이스 목록화 및 추가 권고

### 2차 리뷰 — 품질 리뷰 ("내용이 올바른가?")

**담당 EVAL: 03, 07, 09, 11**

TC 내용의 품질과 정확성에 집중한다. 구조(01, 02 등)는 **보지 않는다** (1차 수정 검증만 수행).

- **1차 수정 반영 확인**: 1차 리뷰 이슈 목록과 1:1 대조하여 수정 반영 여부 명시 (반영/미반영/부분반영)
- **EVAL-03**: 재현 스탭 품질 — F열 추상 표현 금지어 grep: `정상 동작` / `정상적으로` / `올바르게` / `자연스럽게` / `원활히` / `깔끔하게` / `이상 없이` / `문제 없이` / `적절히` / `적당히`. 구체적 기대 결과(화면 변화, 수치 변화, 상태 변화) 포함 여부 확인. **함수적 종속성**: [전제조건]+[행동]이 [기대결과]를 유일하게 결정하는지 확인 — "~할 수도 있다", "~에 따라 다르다" 등 비결정적 기대결과는 조건을 분리하여 별도 TC로 작성해야 함. **"또는" 패턴은 EVAL-15에서 별도 검출**
- **EVAL-07**: 중복 TC 검출 — 완전 중복 / PC+모바일 분리 / 검증단계만 다른 중복
- **EVAL-09**: tc-생성.md 규칙 준수 여부:
  - 1 TC = 1 검증 포인트 위반 ("A되고 B되는지" 형태) — **1NF 위반**: 하나의 TC에서 독립된 2개 이상의 검증을 수행하면 분리 권고
  - 재현 스탭에 플랫폼 중복 기재 ("PC에서", "모바일에서"로 시작)
  - 부정+예외 합계 < 49%
  - 경계값 TC에 구체적 수치 미명시
  - **NULL/빈상태 예외 TC**: tc_design.md의 **[NULL/빈상태 대상 목록]**에서 "O" 표시된 항목을 1:1 대조하여, 해당 빈상태 예외 TC가 실제로 존재하는지 확인. 누락 시 **MEDIUM** 이슈. 목록이 tc_design.md에 없으면 기존 방식(소분류별 빈값 TC 존재 여부)으로 확인
- **EVAL-11**: 취소선/추후구현 교차 검증 — Confluence 기획서 원문에서 취소선·추후구현 항목을 추출하고, TC와 대조:
  - 취소선 항목 기반으로 작성된 TC → 삭제 대상으로 이슈 등록 (**HIGH**)
  - 추후구현 항목 기반 TC인데 비고에 "추후구현" 미표기 → 비고 추가 이슈 등록 (**MEDIUM**)
  - 취소선 뒤 변경된 내용(→ 이후 텍스트)이 있으면, TC가 변경 후 내용을 반영하는지 확인
- **EVAL-13**: 결정 테이블 조합 커버리지 — tc_design.md의 **[결정 테이블]**이 있는 소분류를 대상으로, 각 조건의 단독 실패 케이스가 TC에 존재하는지 확인. 누락 시 **HIGH** 이슈. 결정 테이블이 tc_design.md에 없더라도 기능 설명에 "A이고 B이면" 복합 조건이 있으면 직접 식별하여 확인
  - 동등 분할(EP) 연계 확인: 동일 소분류 내 무효 파티션이 1개로 뭉쳐져 있지 않은지 (서로 다른 실패 이유가 구분되는지) 확인. 뭉쳐 있으면 **MEDIUM** 이슈
- **EVAL-14**: J열(비고) 화이트리스트 — 허용값 5종 외 **모두 HIGH 이슈**. 검출 시 처방: `시트 TC-NNN J열 값 "<검출>" → 빈 값으로 clear`. 단, "추후 구현"·"구현 우선순위 낮음" 키워드가 검출되면 EVAL-17 정합성 함께 확인
- **EVAL-15**: F열 "또는" 분기 검출 — F열에 `또는` 매칭되는 행을 전부 추출하고 4분류로 판정:
  - **결과 분기** (예: "로딩 화면 또는 오류 안내가 표시"): **HIGH**, 처방: `J열에 "기획 확인 필요" 추가`
  - **사전조건 분기** (예: "합성 실패 또는 성공-고급 이하 클릭 시"): **HIGH**, 처방: `두 TC로 분리 (분리 가능 시) 또는 J열에 "기획 확인 필요"`
  - **과정 분기** (예: "Tab 선택 또는 재료 등록 시"): **HIGH**, 동일 처방
  - **허용 상태 나열** (예: "확정 완료 또는 대기 중 하나로 존재"): 결과 1개 + 허용 상태 나열은 통과. 단 기획서에 허용 상태가 명시된 경우만
- **EVAL-16**: 기본기능 섹션 CRITICAL — **J열 `기본기능` 태그가 부여된 행** 단일 검사 (구 규칙 "대분류=기본기능"은 폐기, 2026-04-24부터). 5가지 모두 충족해야 PASS. 1건이라도 미충족 시 **CRITICAL** 처방:
  - ① J열에 `기본기능` 태그 / ② G열 = `PC` / ③ I열 = `N/A` / ④ F열에 GlobalDefine 키 인용 / ⑤ F열에 기획서 안내 문구 큰따옴표 인용
  - 분류(B/C/D)는 일반 TC와 통일된 신규 체계 따름 — 대분류 "기본기능" 발견 시 분류 재작성 처방 (HIGH)
  - G열 ≠ `PC` → 처방: `G열을 "PC"로 변경`
  - I열 ≠ `N/A` → 처방: `I열을 "N/A"로 변경`
  - F열에 GlobalDefine 키 없음 → 처방: `F열에 "(<키명>)" 인용 추가`
  - F열에 기획서 안내 문구 없음 → 처방: `F열에 기획서 정의 안내 문구를 큰따옴표로 인용 추가`
- **EVAL-17**: J열-H/I 정합성 — `추후 구현` 행의 H/I가 N/A 아님 → **HIGH**, 처방: `H/I 모두 N/A로 변경`. `구현 우선순위 낮음`/`기획 확인 필요` 행의 H/I가 N/A → **HIGH** (오처리), 처방: `H/I 모두 미진행으로 변경`. `기본기능` 행의 G≠`PC` 또는 I≠`N/A` → **CRITICAL** (EVAL-16과 연동)
- **EVAL-18**: F열 내부 설계 태그 잔류 검출 — 정규식 `\((ST|DT|C|B)-\d+(\s*#\d+(?:,\s*#?\d+)*)?[^)]*\)` / `\(§\d+-\d+\)` / `` `[A-Za-z_]+`\s*\(미지정\) `` 매칭 행 추출. **HIGH** 이슈. 처방:
  - `(ST-N #M)` / `(DT-N)` / `(B-N)` / `(§N-M)` 같은 보조 표기 → **태그만 제거** (자연어 의미 유지)
  - `(C-N #M)` 또는 `(미지정)` 동반 → **태그 제거 + J열 `기획 확인 필요` + H/I `미진행`**
  - 의미 손실 우려 시 tc_design.md의 해당 태그 정의를 자연어로 인용해 재작성 (처방에 새 F열 초안 포함)

- **EVAL-19**: **분류 계층 적합성 (자동화 사전조건 계층)** — 분류 = 단순 기능 그룹핑이 아닌 자동화 셋업 계층. 4종 위반 검출:
  - ① **대분류가 동사/기능 표현**: `이동 조작`, `합성`, `등록 처리` 등 행위 표현 → **MEDIUM** (카테고리·진입 컨텍스트 표현으로 변경)
  - ② **소분류가 동작 표현 (화면명 아님)**: `재료 등록 클릭`, `삭제 처리`, `이동 중 점프 거리` → **MEDIUM** (현재 화면명으로 변경: `재료 등록 화면`, `이동 중 화면`)
  - ③ **F열에 진입 동작 중복 기재**: 소분류가 이미 "재료 등록 화면"인데 F열에 "HUD에서 합성 버튼 클릭하고 Tab 진입 후 재료 등록 화면에서 ~" → **HIGH** (진입 부분 제거, 화면 위 액션만 남김). 패턴 grep: F열에 `HUD에서`, `~ 버튼을 클릭하고 ~ Tab을 클릭`, `~화면 진입 후 ~화면에서` 등
  - ④ **같은 소분류 TC들의 셋업이 다름**: 한 소분류 안에 셋업이 다른 케이스가 섞임 (예: 신규 계정 + 기존 계정 케이스 혼재) → **MEDIUM** (중분류 분리 권장)
  - 처방 형식: `시트 TC-NNN B/C/D열 "<현재>" → "<제안>" / F열 진입 동작 "<진입문구>" 제거 후 "<액션만>"으로 수정`

### 3차 리뷰 — 최종 검수 ("출시해도 되는가?")

**담당 EVAL: 01~11 전체**

전체 EVAL을 재확인하고, 출시 가능 수준인지 종합 판단한다.

- **2차 수정 반영 확인**: 2차 리뷰 이슈 목록과 1:1 대조하여 수정 반영 여부 명시
- **EVAL 01~11 전체 재확인**: 수정 과정에서 새로 발생한 이슈가 없는지 점검
- **설계 산출물 종합 대조** (tc_design.md에 해당 테이블이 있을 때만):
  - [검증단계 사전 배분표] vs 실제 TC 분포 — 최소 수량 충족 + 부정+예외 49%+ 확인
  - [분류 그룹핑 트리] vs 실제 시트 순서 — 트리 순서와 일치 확인
  - [NULL/빈상태 대상 목록] vs 빈상태 예외 TC — "O" 항목 전수 존재 확인
- 커버리지 100% 달성 여부 최종 판정
- 출시 가능 수준(릴리스 기준)인지 종합 판단
- Pass Gate: EVAL 01~13 전부 PASS + CRITICAL=0 + HIGH=0

### 간이 검증 — 수정 확인 ("2차 지적이 고쳐졌는가?")

2차 리뷰에서 이슈가 있어 수정2를 실행한 경우에만 동작한다. **새로운 이슈는 탐색하지 않는다.**

- 2차 리뷰 보고서(review_[탭명]_v2.md)의 이슈 목록을 읽는다
- 각 이슈에 대해 수정된 TC를 1:1 대조하여 반영 여부만 확인
- 반영 확인 결과: **반영** / **미반영** / **부분반영**
- PASS: 2차 지적 사항 전부 반영 확인 → done
- FAIL: 미반영 건 목록 출력 → done (파이프라인 종료, 수동 확인 필요)

## TC 품질 EVAL 기준

리뷰 시 아래 11개 Eval 항목을 **자기 차수 담당 항목에 한해** 명시적으로 PASS/FAIL 판정하고 보고서에 기록한다.

| Eval | 기준 | 판정 방법 | 담당 차수 |
|------|------|---------|-----------|
| EVAL-01 | 커버리지 100% — 기획서 모든 대/중/소분류가 TC에 존재 | 기획서 분류 목록과 TC 분류 전수 대조 | 1차, 3차 |
| EVAL-02 | 케이스 완전성 — 각 소분류에 정상+부정+예외 최소 1개씩 + 리스크별 부정+예외 비율 (HIGH≥60% / MEDIUM≥49% / LOW≥30%) | 소분류별 검증단계 분포 확인 + 리스크 레벨별 비율 대조 | 1차, 3차 |
| EVAL-03 | 재현 스탭 품질 — 추상적 표현 없음, 구체적 기대 결과 포함, **함수적 종속성**(입력→결과 1:1 결정적) | "정상 동작" 등 검색 + [전제]+[행동]→[결과] 결정성 확인 | 2차, 3차 |
| EVAL-04 | 플랫폼 값 적정 — PC / 모바일 / PC/모바일 외 값 없음 | G열 전체 값 확인 | 1차, **2차(수정본 회귀)**, 3차 |
| EVAL-05 | TC ID 연속성 — 001부터 순서대로, 중복·빠짐 없음 | A열 순서 전수 확인 | 1차, **2차(수정본 회귀)**, 3차 |
| EVAL-06 | 분류 그룹핑 — 동일 소분류 TC가 연속 배치 (분산 금지) | B/C/D열 기준 연속성 확인 | 1차, **2차(수정본 회귀)**, 3차 |
| EVAL-07 | 중복 TC 0건 — 완전 중복, PC/모바일 분리, 검증단계만 다른 경우 | 쌍 비교 검출 | 2차, 3차 |
| EVAL-08 | 분류 표기 — 동일 대/중/소분류 최상단 1회만, 이후 빈칸 | B/C/D열 중복 표기 확인 | 1차, 3차 |
| EVAL-09 | tc-생성.md 규칙 준수 — ①1TC=1검증포인트(**1NF**) ②재현스탭 플랫폼 미중복 ③부정+예외≥49% ④경계값TC 수치 명시 ⑤**NULL/빈상태 예외TC 존재** | 전체 재현스탭 스캔 + 소분류별 빈상태TC 존재 확인 | 2차, 3차 |
| EVAL-10 | 기획서 섹션 전수 대조 + **ER 관계 교차 TC** — 기획서 원문의 모든 섹션 1:1 매핑 + tc_design.md의 [기능 간 관계] 테이블에 명시된 교차 TC 존재 확인 | 기획서 목차 대조 + 관계 매트릭스의 "O" 항목이 TC에 반영되었는지 확인 | 1차, 3차 |
| EVAL-11 | 취소선/추후구현 교차 검증 — Confluence 원문의 취소선(`strike`) 항목 기반 TC가 없고, 추후구현 TC는 비고에 명시 | ADF 취소선 추출 → TC 대조, 추후구현 키워드 검색 → 비고 확인 | 2차, 3차 |
| EVAL-12 | 상태 전이 커버리지 — 상태 있는 소분류의 모든 전이 엣지(화살표)에 TC 존재, 비정상 전이 TC 1건 이상 | tc_design.md [상태 전이 테이블] 또는 직접 식별 → 전이 엣지별 TC 대조 | 1차, 3차 |
| EVAL-13 | 결정 테이블 조합 커버리지 — 복합 조건 소분류에 각 조건 단독 실패 TC 존재 + EP 무효 파티션 구분 | tc_design.md [결정 테이블] 또는 직접 식별 → 조건별 실패 TC 대조, 무효 파티션 뭉침 확인 | 2차, 3차 |
| EVAL-14 | **J열(비고) 화이트리스트** — 허용값(`""` / `기본기능` / `추후 구현` / `구현 우선순위 낮음` / `기획 확인 필요` / `DXBUG-XXXX`) 외 메타정보 0건 | J열 전수 grep → 금지 패턴(`BVA`, `동시성`, `시간조작`, `세션`, `EP 무효`, `기획 변경 사항`, `기획 재확인 필요` 등) 검출 | 2차 |
| EVAL-15 | **F열 "또는" 분기 검출** — 사전조건/과정/결과 어느 하나라도 "또는"으로 분기되면 1NF 위반 | F열 grep `또는` → 매칭 행마다 (사전조건 분기 / 과정 분기 / 결과 분기 / 허용 상태 나열) 4분류 → 분기 발견 시 분리 또는 J=`기획 확인 필요` 처방 | 2차 |
| EVAL-16 | **기본기능 섹션 CRITICAL** — TC-001(또는 기본기능 행) 4가지 모두 충족: G=`PC` / I=`N/A` / F열에 GlobalDefine 키 인용 / F열에 기획서 안내 문구 인용 | 기본기능 행 단일 검증, 4항목 중 1개라도 미충족 시 CRITICAL | 1차, 3차 |
| EVAL-17 | **J열 값별 H/I 정합성** — `추후 구현`→H/I 모두 `N/A`, `구현 우선순위 낮음`/`기획 확인 필요`→H/I `미진행`, `""`→H/I `미진행` (기본기능 제외) | J열 값과 H/I 값 매트릭스 대조 | 2차 |
| EVAL-18 | **F열 내부 설계 태그 잔류** — `(ST-N #M)` / `(DT-N)` / `(C-N #M)` / `(B-N)` / `(§N-M)` / `` `var`(미지정) `` 등 설계 산출물 표기가 시트에 노출됐는지 | F열 정규식 매칭 → 태그 의미 분류(보조 표기 vs 기획 미정 신호) → 처방 차등 적용 | 2차 |
| EVAL-19 | **분류 계층 적합성** — 대=카테고리/진입 컨텍스트, 중=뎁스/셋업, 소=현재 화면명, F열=화면 위 액션만 (자동화 사전조건 계층) | ①대분류 동사 표현 ②소분류 동작 표현 ③F열 진입 동작 중복 ④같은 소분류 셋업 불일치 4종 검출 | 1차 |

### 차수별 통과 기준 (Pass Gate)

| 리뷰 차수 | 담당 EVAL | PASS 조건 | 결과 |
|-----------|-----------|----------|------|
| 1차 (구조) | 01, 02, 04, 05, 06, 08, 10, 12, **16** | 담당 EVAL 전부 PASS + CRITICAL=0 | 수정 불필요 → 2차로 |
| 2차 (품질 + 수정본 회귀) | 03, 07, 09, 11, 13, **14, 15, 17, 18** **+ 04/05/06 회귀** | 담당 EVAL 전부 PASS + 1차 FAIL→PASS 전환 + **수정 1차에서 신규 추가된 TC 범위에 대해 EVAL-04(G열=PC/모바일/PC계열만), EVAL-05(ID 연속), EVAL-06(분류 연속) 재확인** + CRITICAL=0 | 수정 불필요 → 완료 |

> **⚠ 2차 회귀 검사 근거 (2026-04-17 발생 사고)**: 수정1 단계에서 신규 TC 21개가 추가될 때 F열(재현스탭)↔G열(플랫폼) 컬럼이 뒤바뀐 채로 삽입되었으나, 2차 리뷰가 품질 전용이어서 놓쳤음. 이후 파이프라인이 배포까지 진행됨. **수정으로 추가된 TC에 대해 반드시 구조 회귀(EVAL-04/05/06) 실행**.
| 간이 검증 | 2차 FAIL 항목만 | 2차 지적 사항 전부 반영 확인 | done |

> 2차에서 이슈 > 0 → 수정2 실행 → 간이 검증 → done.
> 간이 검증에서 미반영 건이 있으면 목록만 기록하고 done 처리 (파이프라인 종료).

### 리스크 레벨 기반 TC 충실도 체크

tc_design.md의 소분류별 리스크 레벨([HIGH]/[MEDIUM]/[LOW])과 TC 수량/케이스 구성 대조:
- **[HIGH] 소분류**: 부정+예외 케이스가 충분한지 (정상 위주면 HIGH 이슈)
- **[LOW] 소분류**: TC가 과도하게 많지는 않은지 (불필요한 중복 의심)

리스크 레벨 태깅이 tc_design.md에 없으면 **[MEDIUM]** 이슈로 리뷰 보고서에 명시

## 결함 분류 타입 (Defect Classification)

이슈 등록 시 심각도와 함께 **결함 타입 태그**를 1개 병기한다. 반복 패턴 추적 및 수정 우선순위 결정에 활용.

| 결함 타입 | 태그 | 기준 | 해당 EVAL |
|----------|------|------|----------|
| 커버리지 누락 | `[커버리지]` | 기능/섹션/전이 엣지/조건 조합이 TC에 없음 | EVAL-01, 10, 12, 13 |
| 케이스 불완전 | `[케이스]` | 정상/부정/예외 배분 미달, 리스크 비율 미달 | EVAL-02, 09 |
| 재현 스탭 품질 | `[스탭]` | 추상적 표현, 비결정적 기대결과, 수치 미명시 | EVAL-03 |
| 데이터/로직 오류 | `[데이터]` | 수치 처리, 저장 정합성, 롤백 관련 TC 결함 | EVAL-02, 09 |
| 연계/인터페이스 | `[연계]` | 기능 간 교차 TC 누락, 연동 오류 시나리오 없음 | EVAL-10, 13 |
| 상태 전이 | `[상태]` | 전이 엣지 누락, 비정상 전이 TC 없음 | EVAL-12 |
| 구조/서식 | `[구조]` | ID 오류, 분류 분산, 중복 표기, 플랫폼 오기입 | EVAL-04, 05, 06, 07, 08 |
| 기획 불일치 | `[기획]` | 취소선 기반 TC 잔존, 추후구현 미표기, 삭제 기능 TC | EVAL-11 |

**이슈 작성 형식**: `#### [심각도][결함타입] 이슈 제목` — 심각도와 결함타입을 헤더에 함께 표기. TC 번호는 해당 TC가 실제로 존재하는 경우에만 기재하고, 누락 케이스 이슈는 TC 번호 없이 소분류명과 내용으로 특정.

> 결함 타입은 tc-fixer-v2가 수정 우선순위 결정 시 참조한다. `[커버리지]` `[케이스]` `[상태]`가 `[구조]`보다 우선.

### 처방 작성 규칙 (필수)

**tc-fixer는 판단 없이 처방만 실행하는 코더다.** 리뷰어는 이슈마다 fixer가 그대로 실행할 수 있는 처방을 반드시 작성해야 한다.

#### 처방 형식

```
처방: [실행 유형] [대상] [내용]
```

| 실행 유형 | 처방 예시 |
|----------|----------|
| 재현스탭 수정 | `처방: TC-012 F열 수정 — '[현재 재현스탭 전문]' → '[새 재현스탭 1줄 초안]'` |
| 검증단계 변경 | `처방: TC-034 E열 수정 — '부정' → '예외'` |
| 플랫폼 변경 | `처방: TC-056 G열 수정 — 'PC' → 'PC/모바일'` |
| 신규 TC 추가 | `처방: [소분류명] 소분류에 신규 TC 추가 — 검증단계: 예외, 플랫폼: PC/모바일, 재현스탭 초안: '[1줄 초안]'` |
| TC 삭제 | `처방: TC-078 삭제 — 취소선 항목 기반` |
| 비고 추가 | `처방: TC-090 J열 추가 — '추후 구현'` |

> 재현스탭 초안은 완성된 문장이 아니어도 된다. Haiku가 초안을 받아 `[사전조건]+[행동]+[기대결과]` 형식으로 완성한다.
> 처방 없이 `권고: ~하세요` 식으로만 작성하면 fixer가 처리할 수 없다. 반드시 처방 형식을 지킨다.

## 완료 보고 형식

**출력 스타일 (보고서 본문)**: `caveman-ko full` 참조 (`~/.claude/skills/caveman-ko/SKILL.md`).

## 학습 패턴 축적 (리뷰 완료 후 필수)

리뷰 보고서 작성 완료 후, 발견한 이슈가 **기존 규칙 파일(tc-생성.md, tc-수정.md, tc-설계.md)에 이미 예방 규칙이 있는데도 재발한 것인지**, 아니면 **새로운 유형의 반복 이슈인지** 판별한다.

### 판별 절차

1. `tc-학습.md`의 "적용 완료 패턴"에서 이번 이슈와 동일 패턴이 있는지 확인
2. **동일 패턴 재발**: 규칙이 있는데도 재발 → tc-학습.md의 해당 패턴에 `재발 N회` 카운터 추가. 규칙 파일의 해당 항목 강화 검토 필요 표기
3. **신규 패턴 발견**: 2회 이상 동일 유형이 발견된 이슈 → tc-학습.md "미적용 패턴"에 신규 P-번호로 추가

### 신규 패턴 등록 형식

```markdown
### P-XX: [패턴명] `미적용`

- **빈도**: 높음/중간/낮음 (발견 건수: N건, 발견 기능: [기능명1, 기능명2])
- **발생 단계**: tc-writer / tc-fixer / tc-설계
- **증상**: [구체적 증상]
- **원인**: [추정 원인]
- **권고 예방 규칙**: → [어떤 규칙 파일]에 [어떤 내용] 추가 권고
- **적용 대상**: [규칙 파일명] ([어떤 섹션])
```

> **1회만 발견된 이슈는 패턴으로 등록하지 않는다.** 다음 리뷰에서 같은 유형이 재발하면 그때 등록한다.
