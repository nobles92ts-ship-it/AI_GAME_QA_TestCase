# § 1. 실행 가이드 (HOW)

## 1.1 역할

기획서를 읽어 분석하고 **`analysis.md`** 를 생성한다.

> **파일명 규약 (엄수)**:
> - **로컬 specs 경로 파일명**: `analysis.md` / `tc_design.md` — 파이프라인·에이전트가 실제 참조하는 이름
> - **드라이브 업로드 파일명**: `DX-[기능명]_분석_text_1.md` / `DX-[기능명]_tc_design.md` — 사람이 구분용으로 쓰는 별칭
> - 본 문서에서 "analysis.md" = 로컬 파일명, "분석_text_1.md" = 드라이브 별칭. 혼용 금지.

1. 기획서 내용 가져오기 (입력 유형에 따라 처리 방식 다름)
2. 원문을 마크다운으로 정리한 `analysis.md` 작성 (드라이브 업로드 시 별칭: `DX-[기능명]_분석_text_1.md`)
3. 테스트 관점 분석 (EP, BVA, 상태 전이, 결정 테이블) 수행
4. 분석 결과를 tc-설계 단계에 전달할 수 있는 상태로 저장

## 1.3 입력 유형 감지 및 처리

> ⚠️ **파이프라인 모드 우선 규칙**: 핸드오프에 `confluence_raw.md` 파일 경로가 포함되어 있으면, Confluence URL이 있더라도 **절대 `getConfluencePage`를 호출하지 않는다.** `confluence_raw.md`를 Read 도구로 읽어 그것을 기획서 원문으로 사용한다. 팀장(tc-팀-v2)이 이미 저장해둔 파일이다.

| 입력 유형        | 판별 기준                                | 처리 방식                           |
| ---------------- | ---------------------------------------- | ----------------------------------- |
| 파이프라인 파일  | 핸드오프에 `confluence_raw.md` 경로 포함 | Read 도구로 파일 직접 읽기 (최우선) |
| Confluence       | URL에 `atlassian.net` 또는 `/wiki/` 포함 | Atlassian MCP (`getConfluencePage`) |
| PDF              | `.pdf` 확장자 또는 경로                  | `Read` 도구로 직접 읽기             |
| Word (doc/docx)  | `.doc` / `.docx` 확장자                  | `Read` 도구로 직접 읽기             |
| Excel (xlsx/xls) | `.xlsx` / `.xls` 확장자                  | Bash + Node.js로 파싱               |

### Confluence 처리 (confluence_raw.md 없을 때만)

- Atlassian MCP(`getConfluencePage`)로 **ADF 형식**으로 가져오기
  - cloudId: `{CONFLUENCE_SITE}`
  - pageId: URL 마지막 숫자
- **취소선(`strike` mark) 포함 모든 서식 누락 없이 확인**
  - 취소선 텍스트는 반드시 `~~텍스트~~` 형식으로 변환
  - 링크, 볼드, 이탤릭 등 서식도 그대로 반영

### PDF / doc / docx 처리

1. Read 도구로 파일 읽기
2. 기능명: 파일명에서 추출 (확장자 제거, 공백→`_`, 특수문자 제거)
3. 읽은 내용을 그대로 원문으로 사용 → `analysis.md` 작성

### Excel (xlsx/xls) 처리

```bash
{NODE_PATH} -e "
const XLSX = require('xlsx');
const wb = XLSX.readFile('[파일경로]');
wb.SheetNames.forEach(name => {
  const csv = XLSX.utils.sheet_to_csv(wb.Sheets[name]);
  console.log('=== Sheet: ' + name + ' ===');
  console.log(csv);
});
"
```

> xlsx 모듈 없을 시: `cd {WORK_ROOT} && npm install xlsx` 후 재실행.

### 기능명 추출 규칙

1. **파일명 우선**: 확장자 제거 후 공백→`_`, 특수문자 제거
2. **내용에 제목 있을 시**: 문서 첫 줄/헤딩이 있으면 그것을 기능명으로 사용
3. 불명확한 경우 파일명 그대로 사용

## 1.4 이미지 분석 (텍스트 선행 2-pass 방식)

Confluence 이미지는 `confluence_image_downloader.py`로 다운로드한 후, **텍스트 분석 결과를 컨텍스트로 삼아** 이미지를 해석한다. 이미지 단독 분석 금지 — 텍스트가 만든 Part A 섹션에 이미지를 맞춰 보완하는 방식.

**Pass 1 (다운로드 + 텍스트 작성)**: 이미지를 Read하지 않는다. 다운로더가 `images/img_01.png`, `img_02.png`, ... 순번 파일로 저장. 텍스트 기반으로 Part A/B/C를 작성하되, 이미지가 등장하는 모든 위치에 플레이스홀더 삽입:
```markdown
> [IMG: img_01.png] (보류 — Pass 2에서 채움)
```

**Pass 2 (컨텍스트 기반 이미지 분석)**: 각 플레이스홀더에 대해 앞뒤 ±500자 텍스트(= 문맥)를 추출하고 `Read(images/img_NN.png)`로 이미지 로드. 텍스트에 이미 설명된 내용은 **재추출 금지**, 텍스트에 없는 시각 정보(UI 배치/색상/아이콘/연출/수치 라벨)만 보완. 플레이스홀더를 아래 형식으로 치환:
```markdown
> [IMG: img_01.png] — 섹션: A-2 캐릭터 선택 UI
> - 시각 보완: {텍스트에 없는 정보만}
> - 원본: images/img_01.png
```

**실패/특수 케이스**:
- 다운로드 실패 → `[IMG: img_NN — 다운로드 실패, 원본 확인 필요]`
- 분석 실패(손상/판독 불가) → `[IMG: img_NN.png — 분석 불가]`
- alt에 `.gif`/`.mp4` 포함 → `[IMG: img_NN.gif — 연출 참조 필요, 정적 스냅샷만 분석]`

> **설계 반영**: 위 3개 실패/특수 케이스가 포함된 소분류는 tc_design.md에서 `[이미지 참조 필요]` 태그 부여 (설계 Step 6 "이미지 플레이스홀더 스캔"에서 자동 처리).

## 1.5 작업 흐름 (Step 1~8)

1. **상태 갱신 (시작)**: `echo "분석중|[기능명]" > status/tc-designer.txt`
2. **입력 유형 판별** → Confluence/PDF/doc/Excel 기준 처리 (§1.3 참조)
   - **파이프라인 모드**: `confluence_raw.md` 파일 존재 시 직접 읽기 (MCP 재호출 금지)
3. **기능명 추출** (공백→`_`, 특수문자 제거)
4. **이미지 다운로드만 실행** (이미지 존재 시):
   - `confluence_image_downloader.py` 실행 → `images/img_01.png`, `img_02.png`, ... 순번 파일명으로 저장
   - **이 단계에서 이미지를 Read로 읽지 않는다** (Pass 2에서 처리)
   - 다운로드 실패 이미지는 목록에 기록 → Pass 1에서 `[IMG: img_NN — 다운로드 실패]`로 표기
5. **기존 파일 확인**: `ls "$SPECS/[기능명]/analysis.md"` — 있으면 재사용, 없으면 신규 작성. 드라이브 측 `DX-*_분석_text_1.md` 존재 시에도 로컬 analysis.md 재생성.
6. **Pass 1 — 텍스트 기반 1차 작성** (상세: §1.4 Pass 1 참조):
   - 메타데이터 → Part A(플레이스홀더 삽입, 이미지 Read 금지) → Part B(B-1~B-8) → Part C(C-1, C-2) 순 append
   - 메타데이터의 `이미지 개수`는 `N장 (분석 완료: 0장)` 초기 기록
   - 모두 작성 후 파일 1차 저장
7. **Pass 2 — 이미지 플레이스홀더 치환** (상세: §1.4 Pass 2 참조):
   - 각 `[IMG: img_NN.png]`에 앞뒤 ±500자 문맥 추출 + `Read(images/img_NN.png)` → 텍스트에 없는 시각 정보만 Edit 치환
   - 완료 후 메타데이터 `이미지 개수` → `N장 (분석 완료: M장)` 업데이트 + 최종 저장
8. **자체 검증 체크리스트 실행** (§1.6, 설계 단계 시작 전 필수)

## 1.6 자체 검증 체크리스트 (Step 8)

아래 항목을 모두 통과해야 분석 단계 완료로 간주. **하나라도 실패 시 해당 Part 재작성 또는 Pass 2 재실행.**

```markdown
[메타데이터]
[ ] 기획서 URL / 해시 / 기능명 / 분석일시 기록됨
[ ] 이미지 개수(총/완료) 표기됨 — Pass 2 완료 후 `분석 완료: M장` 업데이트됨

[Part A — 원문 정리본]
[ ] A-1 개요 작성됨
[ ] A-2 기능 정의 — 기획서의 모든 상위 섹션 반영
[ ] A-3 이미지 분석이 본문 플레이스홀더 자리에 인라인 치환됨
[ ] `[IMG: img_NN.png] (보류 — Pass 2에서 채움)` 미치환 플레이스홀더가 남아 있지 않음 (다운로드 실패/분석 불가 명시는 허용)
[ ] 각 이미지 분석 블록에 "섹션: A-N …" 명시 있음 (어느 섹션 보완인지 표기)
[ ] A-4 토스트/팝업 메시지 표 존재
[ ] A-5 GlobalDefine 키 표 존재
[ ] A-6 취소선/추후구현 항목 ("없음" 명시도 허용)

[Part B — 테스트 분석 레이어]
[ ] B-1 용어 사전 (기능 고유 용어가 있을 때만 작성, 없으면 "해당 없음")
[ ] B-2 외부 의존성 표 존재
[ ] B-3 EP 파티션 후보 — 수치/상태 조건 있는 모든 소분류 포함
[ ] B-4 BVA 후보 수치 — 모든 수치 GlobalDefine 키 포함
[ ] B-5 상태 머신 후보 목록 — 상태 있는 소분류 전수 식별
[ ] B-6 결정 테이블 후보 목록 — 복합 조건 소분류 전수 식별
[ ] B-7 암묵적 태그 매핑 — **모든 소분류** 전수 나열 (빈 셀 금지)
[ ] B-8 오류 패턴 매칭 — 각 소분류가 오류 카탈로그 행과 대조됨

[Part C — 설계 힌트]
[ ] C-1 미지정값 재확인 체크리스트 — `N (미지정)` 패턴 전수 추출
[ ] C-2 분석자 코멘트 — 없으면 "모호 지점 없음" 명시

[품질]
[ ] 어느 섹션도 생략되지 않음 (해당 없음도 명시)
[ ] Part B의 "후보"와 tc_design.md의 "확정 테이블" 역할이 중복되지 않음
```

> **실행 방법**: Step 7 완료 직후 Opus가 self-review로 위 체크리스트를 돌리고, 실패 항목 발견 시 해당 Part만 재작성. 통과 후 설계 단계(tc-설계.md) 시작.

# § 2. 출력 스펙 (WHAT)

## 2.1 MD 파일 작성 규칙

### 파일명 형식

**로컬 specs 폴더** (파이프라인·에이전트 참조용):
- `analysis.md` — 기획서 원문 정리본 + 테스트 분석 레이어 + 설계 힌트
- `tc_design.md` — TC 작성용 기능 구조 분석본 (tc-설계 단계에서 생성)

**드라이브 업로드 파일명** (사람 구분용 별칭):
- `DX-[기능명]_분석_text_1.md` ↔ `analysis.md`
- `DX-[기능명]_설계_text_2.md` ↔ `tc_design.md`

**[기능명] 규칙**: Confluence 페이지 제목에서 추출. 공백→`_`, 특수문자 제거, 한국어 그대로 사용.
예: 드라이브 업로드 시 `DX-회복_물약_퀵슬롯_시스템_분석_text_1.md`

## 2.2 analysis.md 필수 구조 (템플릿 강제)

> ⚠️ **CRITICAL**: 3개 Part 전부 작성 필수 (Part A만 쓰면 검증 FAIL). 각 섹션 순서 준수, 해당 없으면 `"해당 없음"` 명시.

### Part 경계 원칙 (중복 방지)

| Part | 역할 | 책임 |
|------|------|------|
| **Part A** | 원문 정리본 | 기획서 내용을 마크다운으로 보존 |
| **Part B** | 테스트 분석 레이어 | **"후보 식별"** — EP/BVA/상태/결정/태그 후보만 발견 |
| **Part C** | 설계 힌트 | 설계 단계(tc-설계.md)로 넘기는 초안/코멘트 |

**중복 방지 규칙**:
- analysis.md는 **"후보 목록"**까지만 작성 (실제 TC/매핑표/배분표는 설계 단계 책임)
- tc_design.md는 analysis.md의 후보를 받아 **"확정 테이블"**로 변환 (커버리지 매핑표, 검증단계 배분표, 상태 전이 실제 엣지, 결정 테이블 실제 케이스)
- 동일 정보를 두 파일에 복제하지 말 것. analysis = "What exists", design = "What to test"

## 2.4 Part A — 원문 정리본

**A-1. 개요** — 문서 목적, 주요 포인트
**A-2. 기능 정의** — 기획서 내용을 마크다운으로 정리 (원문 최대 보존)
  - 취소선: `~~텍스트~~` 변환
  - 플랫폼별 조작: **모바일** / **PC** 볼드 표기
  - 외부 링크, 미완성 메모, 플레이스홀더 그대로 보존
**A-3. 이미지 분석 통합** — Pass 2에서 각 이미지 플레이스홀더 자리에 인라인 치환 (별도 파일 분리 금지). 이미지는 본 섹션에서 독립적으로 나열하지 않고, A-1/A-2 본문 흐름 속 원래 위치에 배치된 플레이스홀더가 채워지는 방식.
**A-4. 토스트/팝업 메시지 목록** — `키 | 문구 | 발생 상황` 표
**A-5. GlobalDefine 키 목록** — `키 | 초기값 | 용도` 표
**A-6. 취소선/추후구현 항목** — 원문에서 추출 (없으면 "없음" 명시)

## 2.5 Part B — 테스트 분석 레이어 (후보 식별)

> 이 Part는 **후보 목록**만 작성한다. 실제 테이블/매핑은 tc-설계.md가 담당.

**B-1. 용어 사전** — 기능 고유 용어 표. 컬럼: `용어 | 정의 | 동의어/관련`. 없으면 "해당 없음".

**B-2. 외부 의존성 & 연계 기능** — 외부 시스템/데이터/연계. 컬럼: `# | 의존 대상 | 유형 | 등장 섹션 | 관계 유형 (1:1/1:M/M:N)`

**B-3. EP 파티션 후보** (동등 분할) — 수치/상태/입력 조건 있는 소분류에 유효/무효 파티션 식별. 유효는 1건 대표, 무효는 **실패 사유별**로 분리. 컬럼: `# | 소분류 | 입력/조건 | 유효 파티션 | 무효 파티션 (분리 사유)`

**B-4. BVA 후보 수치** (경계값 분석) — 수치 제한 소분류의 4포인트 (`min-1` / `min` / `max` / `max+1`). 실제 TC는 설계 단계에서 작성. 컬럼: `# | 소분류 | 수치 출처 (GlobalDefine 키) | 현재값 | min-1 / min / max / max+1`. 참조: §3.3 경계값 참조표

**B-5. 상태 머신 후보 목록** — 상태 있는 소분류 식별 + 기본 상태/트리거 스케치. 실제 전이 테이블은 tc-설계.md (EVAL-12)에서 완성.
- 대상 판별 기준: 활성/비활성 토글, 시간 기반 전이(쿨타임/카운트다운), 진행 단계(퀘스트/강화), 자원 소모(내구도/스택), 잠금/해제
- 컬럼: `# | 소분류 | 상태 종류 | 주요 트리거 | 복잡도`

**B-6. 결정 테이블 후보 목록** — 복합 조건(2개 이상 AND/OR) 소분류 식별. 실제 매트릭스는 tc-설계.md (EVAL-13)에서 완성.
- 대상 판별 기준: "A이고 B이면", "A이거나 B이면", 복수 조건 동시 체크
- 컬럼: `# | 소분류 | 조건 개수 | 조건 목록 | 결정 결과 유형`

**B-7. 암묵적 요구사항 태그 매핑** — **모든** 소분류 전수 나열 + 태그 부여. 빈 셀 금지 ("해당 없음" 명시). 컬럼: `소분류 | 적용 태그 | 근거 (한 줄)`. 태그 카탈로그: §3.1 참조
> 태그 후보: `[세션] [권한] [데이터] [UI] [유저상태] [동시성] [대소문자] [시간조작] [연계]`

**B-8. 오류 패턴 매칭 (Error Guessing)** — 각 소분류를 §3.2 "오류 추정 카탈로그" 행과 대조. 컬럼: `소분류 | 매칭 기능 유형 | 반영 필요 오류 패턴 | 비고`
> **설계 반영 필수**: 각 행은 tc_design.md "분류 그룹핑 트리"의 해당 소분류 `→ 부정/예외` 케이스로 승격. 설계 Step 6에서 B-8 스캔 후 반영 누락 시 케이스 보강.

## 2.6 Part C — 설계 힌트 (분석 고유 정보만)

> 리스크/TC 배분은 설계 단계가 원문 기반으로 직접 결정하므로 분석에서 초안을 만들지 않는다. Part C는 **설계가 원문만 읽어선 알 수 없는 정보**만 담는다.

**C-1. 미지정값 재확인 체크리스트** — `N (미지정)`, `TBD`, `시간 (미지정)` 전수 추출. 컬럼: `# | 키/항목 | 등장 위치 | 현재 상태 | 재확인 필요 여부`
> **설계 반영 필수**: 모든 항목은 tc_design.md "GlobalDefine 키 목록" `미지정` 컬럼에 `⚠` 표기 + 메타데이터 바로 아래 **"기획 확인 필요 항목"** 블록으로 승격. tc-writer는 해당 키 참조 TC에서 구체 수치 대신 키 이름(예: `Character_DeleteLevelLimit`)으로 재현스탭 작성 (값 결정 시 TC 수정 없이 GlobalDefine만 갱신).

**C-2. 분석자 코멘트 (모호/불명확 지점)** — Opus가 발견한 해석 모호 지점 기록. 1차/2차 리뷰 이슈 근거. 컬럼: `# | 섹션 | 모호 지점 | 가정 | 리뷰 확인 필요`

# § 3. 참조 데이터 (LOOKUP)

## 3.1 암묵적 요구사항 태그 카탈로그

기획서에 명시되지 않아도 아래 카테고리를 기능별로 **해당 여부만 판단**하여, 해당 시 tc_design.md의 소분류에 **[암묵적 요구사항]** 태그로 표기한다. 세부 TC 항목은 tc-writer가 작성.

| 카테고리 | 해당 여부 판단 기준 | 태그 |
|----------|---------------------|------|
| 세션 연속성 | 앱 백그라운드 전환, 네트워크 끊김, 서버 점검, 세션 만료 영향 있는 기능 | `[세션]` |
| 권한/조건 미충족 | 레벨·퀘스트·재화 등 선행 조건이 있는 기능, 역할별 권한 차이 | `[권한]` |
| 데이터 일관성 | 수치 변동(재화·스탯·수량), 연속 실행 가능, 저장 정합성 필요 | `[데이터]` |
| UI 일관성 | 팝업/모달 닫기, 로딩 중 조작, 실패 롤백 | `[UI]` |
| 사용자 상태 예외 | 제재/차단 대상, 탈퇴/삭제 유저, 계정 정보 변경 (소셜/관계 기능) | `[유저상태]` |
| 동시성 (ACID) | 다단계 처리(강화·제작·거래), 동시 요청 가능, 재화/아이템 이동 | `[동시성]` |
| 입력 대소문자 | 텍스트 입력창이 있고 영문 입력 가능한 경우 — 한글 전용이면 해당 없음. 대소문자 구분 여부, 반대 입력 시 처리 확인 | `[대소문자]` |
| 로컬 시간 조작 | 기능이 로컬 타임 기반 시간 처리를 포함하는 경우 (대기 시간, 카운트다운, 갱신 주기 등) — 디바이스/PC 시간 강제 조정 시 반영되지 않는지 확인 | `[시간조작]` |
| 기능 간 관계 | 다른 시스템과 1:1/1:M/M:N 관계, 선행 조건, 연동 기능 | `[연계]` |

**tc_design.md 작성 형식**: 해당 소분류에 태그 추가

```markdown
- 친구 삭제 [HIGH] [PC/모바일] [데이터] [유저상태]
```

**기능 간 관계**가 있으면 tc_design.md에 별도 정리 (상태 전이·결정 테이블 섹션 이후):

```markdown
### 기능 간 관계
| 연관 기능 | 관계 유형 | 교차 TC 필요 여부 | 비고 |
| --------- | --------- | ----------------- | ---- |
| 인벤토리  | 1:M       | O — 0건/1건/최대건 | 아이템 획득 시 |
```

## 3.2 오류 추정 카탈로그 (Error Guessing)

기능 유형별로 자주 누락되는 오류 패턴 목록. tc_design.md 작성 시 소분류를 기능 유형에 대조하여, 해당 패턴이 이미 반영됐는지 확인한다.

| 기능 유형 | 연관 암묵적 요구사항 태그 | 자주 누락되는 오류 패턴 | tc_design.md 반영 방법 |
|---------|---------------------|------------------|-------------------|
| 수량/재화 변경 | `[데이터]` `[동시성]` | 동시 요청 중복 소모, 음수/0 입력, 롤백 없는 실패, 최대치 초과 소모 | `[동시성]` 태그 + BVA 4포인트 |
| 상태 전이 | `[세션]` `[데이터]` | 중간 상태에서 앱 강제 종료, 네트워크 끊김 후 상태 불일치, 동일 이벤트 재발생(비정상 전이) | 상태 전이 테이블 작성 + 예외 행 포함 |
| 시간 기반 기능 | `[시간조작]` `[세션]` | 클라이언트 시간 조작, 자정/주간 갱신 경계, 앱 백그라운드 중 시간 경과 | `[시간조작]` 태그 + 경계 시점 TC 명시 |
| 조건부 잠금/해제 | `[권한]` `[연계]` | 조건 변경 직후 캐시 미반영, 복수 조건 중 하나만 충족, 조건 충족 후 재잠금 여부 | 결정 테이블 작성 + `[권한]` 태그 |
| 목록/검색/페이지네이션 | `[UI]` `[데이터]` | 빈 목록(0건), 최대 항목 수 도달, 마지막 페이지 경계, 정렬 기준 변경 후 유지 여부 | NULL/빈상태 대상 목록 + BVA |
| 텍스트 입력 | `[대소문자]` `[데이터]` | 최대 길이 초과, 특수문자/이모지 입력, 공백만 입력, 복사-붙여넣기 | BVA 최대 길이 + `[대소문자]` 태그 |
| 연동/인터페이스 | `[연계]` `[데이터]` | A 기능 완료 후 B 화면 미반영, 연동 기능 삭제 후 참조 오류, 순서 의존성 역전 | 기능 간 관계 테이블 + 교차 TC 명시 |

**사용 방법**:
1. tc_design.md 소분류 작성 시 위 표에서 해당 기능 유형 행을 찾아 대조
2. 오류 패턴이 소분류 설명 또는 검증단계 배분표에 반영됐는지 확인
3. 반영 안 됐으면 해당 소분류에 추가 (부정/예외 케이스로 보강)

> 암묵적 요구사항 태그와 1:1 연결됨 — 태그가 붙은 소분류는 반드시 해당 행의 오류 패턴을 검토.

## 3.3 경계값 참조표

수치 조건이 있는 기능 분석 시 아래 표를 기준으로 경계값 EP/BVA 포인트를 도출한다.

| 대상          | 경계값 케이스                                        |
| ------------- | ---------------------------------------------------- |
| 마나/MP       | 소모량-1 (부정) / 정확히 소모량 (예외)               |
| HP            | 1 생존 최소 / 0 사망 / 최대HP                        |
| 골드/재화     | 구매가-1 (부정) / 정확히 구매가 (예외) / 최대 보유량 |
| 아이템 보유량 | 0 / 최대 / 최대+1 초과 시도                          |
| 스택/중첩     | 최대-1 / 최대 / 최대+1 시도                          |
| 쿨타임        | 쿨타임 종료 직전 / 정확히 0                          |
| 수량 입력     | 1 / 최대 / 최대+1                                    |

> TC 재현 스탭에 구체적 수치 필수: "마나 49인 상태에서 소모량 50의 스킬 사용 시 시전이 차단되는지 확인"
